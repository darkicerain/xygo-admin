# XYGo 前端架构升级方案（BuildAdmin 风格）

> 参照 BuildAdmin 的目录设计，采用"功能分层 + 插件混入"模式

## 1. 总体目标

- **双层分离**：后台(backend) / 前台(frontend) 功能分层
- **插件市场**：插件独立开发，安装时复制到项目目录
- **保留优势**：保留 Art 组件体系、路由守卫、动态菜单
- **兼容 GoFrame**：API 路径规范与 GoFrame 一致

## 2. 目标目录结构

```
src/
├── api/
│   ├── backend/           # 后台 API（系统核心 + 已安装插件）
│   │   ├── system/        # 系统管理
│   │   ├── auth/          # 认证相关
│   │   └── [plugin]/      # 插件安装后混入
│   └── frontend/          # 前台 API
│       └── [plugin]/
├── views/
│   ├── backend/           # 后台页面
│   │   ├── system/        # 系统管理
│   │   ├── auth/          # 认证相关
│   │   └── [plugin]/      # 插件安装后混入
│   └── frontend/          # 前台页面
│       └── [plugin]/
├── stores/
│   ├── backend/           # 后台状态
│   └── frontend/          # 前台状态
├── router/
│   ├── backend/           # 后台路由
│   └── frontend/          # 前台路由
└── modules/               # 插件包存放目录
    ├── [plugin-name]/     # 未安装的插件包
    │   ├── info.json      # 插件信息
    │   ├── api/
    │   ├── views/
    │   ├── store/
    │   └── routes/
    └── backups/           # 卸载备份
```

## 3. 核心设计理念

### 3.1 系统核心功能 vs 插件

| 类型 | 位置 | 说明 |
|------|------|------|
| 系统核心 | `src/views/backend/system/` | 用户、角色、菜单、部门等 |
| 插件功能 | `src/modules/[name]/` → 安装后混入 | 微信、支付、CMS 等扩展 |

### 3.2 插件安装机制

```
安装流程：
1. 下载插件包到 src/modules/[plugin-name]/
2. 读取 info.json 获取插件信息
3. 将 api/ 复制到 src/api/backend/[plugin-name]/
4. 将 views/ 复制到 src/views/backend/[plugin-name]/
5. 将 routes/ 合并到 src/router/backend/
6. 执行 install.sql（如有）
7. 更新插件状态表
```

### 3.3 插件卸载机制

```
卸载流程：
1. 备份插件文件到 src/modules/backups/
2. 删除 src/api/backend/[plugin-name]/
3. 删除 src/views/backend/[plugin-name]/
4. 从路由中移除插件路由
5. 执行 uninstall.sql（如有）
6. 更新插件状态表
```

## 4. 与 BuildAdmin 的对应关系

| BuildAdmin | XYGo | 说明 |
|------------|------|------|
| `web/src/api/backend/` | `src/api/backend/` | 后台 API |
| `web/src/views/backend/` | `src/views/backend/` | 后台页面 |
| `modules/` | `src/modules/` | 插件包目录 |
| `info.ini` | `info.json` | 插件信息（改用 JSON 格式） |

## 5. API 分层规范

```typescript
// src/api/backend/system/user.ts
import { adminRequest } from '@/utils/request'

export function getUserList(params) {
  return adminRequest.get('/admin/user/list', { params })
}

// src/api/frontend/article/list.ts
import { siteRequest } from '@/utils/request'

export function getArticleList(params) {
  return siteRequest.get('/site/article/list', { params })
}
```

## 6. 迁移路径

### 阶段 1：目录重构
- 创建 `src/api/backend/`、`src/views/backend/` 等目录
- 将现有系统功能迁移到新目录
- 更新路由和导入路径

### 阶段 2：插件系统
- 创建 `src/modules/` 插件目录
- 开发插件管理器（PluginManager）
- 开发插件管理界面

### 阶段 3：插件市场
- 后端提供插件市场 API
- 前端实现插件下载、安装、卸载
- 插件版本管理与升级

## 7. 风险与约束

- 迁移期间需批量修改导入路径
- 路由需支持动态加载插件路由
- 插件安装/卸载需要完善的文件操作

## 8. 结论

采用 BuildAdmin 风格的"功能分层 + 插件混入"模式：
- 系统核心功能在主目录，结构清晰
- 插件独立开发，安装时混入项目
- 符合 BuildAdmin 用户习惯
- 适配 GoFrame 后端
