### 权限开发任务拆解（基于《权限设计文档》）

> 目标：在现有用户/角色/登录能力基础上，按阶段完成菜单、角色授权、接口鉴权、数据权限等功能，实现一套可复用的后台 RBAC 能力。  
> 说明：本文只列“权限相关”的前后端任务，其他模块（用户管理、部门、岗位等）仍按《开发规划文档》执行。

---

### 1. 当前已完成的基础能力（不再重复开发）

- **后端**
  - 登录与会话：`/admin/auth/login`、`/admin/auth/logout`、`/admin/auth/profile`。
  - JWT + 单点登录：`internal/token` + 文件会话存储。
  - 统一返回结构：`middleware.ResponseHandler` → `{ code,message,data,traceId }`。
  - 通用错误码：`internal/consts/code.go`。
  - 鉴权中间件（登录态）：`middleware.AdminAuth`，保护 `/admin/**`。
  - 基础 RBAC 表：`xy_admin_user / xy_admin_role / xy_admin_user_role / xy_admin_menu / xy_admin_role_menu / xy_admin_dept / xy_admin_dept_closure`（已建表）。
  - 角色列表 + CRUD：`/admin/role/list/detail/save/delete`（含 `data_scope/custom_depts` 字段）。
  - Profile 返回角色编码：`/admin/auth/profile` 返回 `roles: []string`，与 `xy_admin_role.key` 对齐。
- **前端**
  - art-design-pro 工程接入，并已使用真实后端登录 `/admin/auth/login`。
  - Axios 封装适配：`code=0` 成功、`code=61` 未登录，CORS 处理。
  - 登录页使用真实接口，登录成功后进入首页。
  - 角色管理页（列表 / 搜索 / 新增 / 编辑 / 删除）已对接 `/admin/role/**`。

### 2. 权限开发阶段划分（总览）

1. **阶段 A：菜单管理与菜单树**
   - 设计并实现菜单增删改查，支持目录/菜单/按钮三种类型。
   - 后端提供菜单树与路由树接口；前端菜单管理页与路由动态注入改为调用后端。
2. **阶段 B：角色–菜单授权（菜单权限）**
   - 角色授权界面接上菜单树，可勾选/保存角色可见的菜单与按钮。
   - 后端维护 `xy_admin_role_menu`，保证不同角色菜单差异化。
3. **阶段 C：接口级权限点（METHOD + PATH）**
   - 菜单按钮配置权限点；后端实现 `PermissionMiddleware`，按 `METHOD + PATH` 校验接口权限。
   - `/admin/auth/profile` 或专用接口返回当前用户所有权限点，前端按钮控制统一用权限点判断。
4. **阶段 D：数据权限（行级）**
   - 角色 `data_scope/custom_depts` 对接前后端；部门闭包表维护对数据权限生效。
   - 选定 1~2 个业务列表（如用户列表、操作日志）作为样板，实现行级数据权限。
5. **阶段 E：权限管理体验优化**
   - 支持复制角色权限、一键继承上级角色菜单与数据范围。
   - 提供权限变更日志（可选）。

下文按阶段拆成可执行的具体任务，每一条都给出“后端 / 前端 / 验收”三部分。

---

### 3. 阶段 A：菜单管理与菜单树

#### A-1. 设计并固化菜单表字段（对齐现有 `xy_admin_menu`）

- **后端任务**
  - 在《权限设计文档》中已有字段列表的基础上，补充必要注释与枚举含义：
    - `type`（1目录 / 2菜单 / 3按钮）。
    - `perms`（JSON 数组，权限点列表，元素为 `METHOD + PATH`）。
    - `hidden/affix/keep_alive/always_show` 等前端相关字段。
  - 在 `internal/model/entity/admin_menu.go` 与 `model/input/adminin/menu.go` 中补齐：
    - 菜单编辑入参 `AdminMenuEditInp`（对应目录/菜单/按钮所需字段）。
    - 菜单树模型 `MenuTree`（`children` 嵌套）。
- **前端任务**
  - 在菜单管理页（`web/src/views/system/menu`）确认所需字段与交互（目录/菜单/按钮）。
  - 将字段命名与后端约定对齐（避免前后端字段不一致）。
- **验收**
  - 数据库层：`xy_admin_menu` 字段注释清晰，能完整表达目录/菜单/按钮的含义。
  - Go 模型层：`entity` 与 `adminin` 的字段与表结构一致、带中文注释。

#### A-2. 实现菜单增删改查接口

- **后端任务**
  - 新增接口（命名可微调）：
    - `GET /admin/menu/tree`：返回完整菜单树（用于管理页面）。
    - `GET /admin/menu/detail`：按 ID 查详情。
    - `POST /admin/menu/save`：新增/编辑菜单或按钮。
    - `POST /admin/menu/delete`：软删/硬删菜单（考虑有子节点时的处理策略）。
  - 控制器：`internal/controller/admin/menu.go`。
  - 模型输入输出：`internal/model/input/adminin/menu.go`。
  - 注意：
    - 维护 `pid/level/tree` 字段（可参考 HotGo 的树维护逻辑）。
    - 新增/修改菜单时，校验父节点合法性、名称/路由唯一性等。
- **前端任务**
  - 调整菜单管理页：
    - 列表数据改为使用 `/admin/menu/tree`。
    - 新增/编辑/删除操作改为调用 `/admin/menu/save/delete`。
  - 确保 UI 支持新增按钮类型（type=3），并能配置权限点 `perms`。
- **验收**
  - 通过前端菜单管理页面，可以：
    - 新增目录/菜单/按钮。
    - 编辑菜单属性。
    - 删除菜单（有子菜单时行为符合预期）。
  - DB 中 `pid/level/tree` 字段正确维护，树结构无断层。

#### A-3. 提供当前用户可见菜单路由接口

- **后端任务**
  - 新增接口：`GET /admin/menu/routes`：
    - 根据当前登录用户的 `roles` → 角色–菜单关系 → 菜单树，返回只包含“有权限菜单”的路由树。
    - 节点包含：`name/path/component/meta{title,icon,hidden,affix,...}`。
  - 放在 `internal/controller/admin/menu.go` 中，复用菜单模型。
- **前端任务**
  - 待办：在 `web/src/api/*` 增加 `fetchMenuRoutes` 请求 `/admin/menu/routes`；当前先不接入动态路由，收口时改造：
    - 登录后调用 `/admin/menu/routes` 获取菜单树。
    - 动态注册路由、生成侧边菜单。
  - 确认与 art-design-pro 的动态路由加载机制兼容。
- **验收**
  - 不同角色登录后，看见的左侧菜单树不同（符合角色–菜单绑定关系）。
  - 刷新浏览器后，菜单仍由后端数据驱动，路由跳转正常。

### 4. 阶段 B：角色–菜单授权（菜单权限）

#### B-1. 角色授权接口与数据结构

- **后端任务**
  - 接口：
    - `GET /admin/role/menuIds`：按角色返回已绑定的 `menu_id` 列表。
    - `POST /admin/role/bindMenus`：为指定角色保存菜单 ID 列表。
  - 读写表：`xy_admin_role_menu`。
  - 需考虑：
    - 清空再写 vs 差异更新策略（第一版可采用清空再插入）。
    - 仅允许编辑非系统内置角色的菜单（可选）。
- **前端任务**
  - 角色管理页的“菜单权限”弹窗（`role-permission-dialog.vue`）：
    - 打开时调用 `role/menuIds` + `menu/tree`，勾选已授权菜单。
    - 保存时调用 `bindMenus`。
  - 支持树形多选、全选/反选等操作（组件已有雏形）。
- **验收**
  - 为某个角色勾选一部分菜单 → 重新登录该角色账号，只能看到勾选菜单（配合 A-3 的路由接口）。

#### B-2. 超级管理员/系统内置角色策略

- **后端任务**
  - 在 `xy_admin_user` 或 `xy_admin_role` 中保留“超级管理员标记”（当前有 `is_super`）。
  - 约定：
    - 超级管理员不受角色–菜单绑定限制，拥有所有菜单与接口权限。
  - 在菜单路由接口、权限中间件等处增加 `isSuper` 快捷判断。
- **前端任务**
  - 如有必要，在角色管理页标识系统内置角色，禁止删除或限制编辑。
- **验收**
  - 超级管理员账号登录时，菜单完整，权限校验恒通过。
  - 非超级管理员受到角色–菜单限制。

### 5. 阶段 C：接口级权限点（METHOD + PATH）

#### C-1. 菜单配置权限点字段

- **后端任务**
  - 确认 `xy_admin_menu` 中用于存储权限点的字段（如 `perms`），类型为 JSON 数组。
  - 对于按钮类型菜单（type=3），在编辑接口中校验 `perms` 字段格式：
    - 每个元素符合 `"METHOD /path"` 模式。
  - 提供一个工具函数：`NormalizePermission(method, path)`，用于统一大小写和路径模板（如 `:id`）。
- **前端任务**
  - 菜单编辑弹窗中，为按钮类型增加“权限点配置”区域：
    - 支持简单的增删行，每行输入 `METHOD` + `PATH`。
  - 表单验证：避免用户输入奇怪的权限点格式。
- **验收**
  - 在菜单管理页面可以为按钮配置多个权限点，保存后 DB 中 `perms` 字段为 JSON 数组且内容合法。

#### C-2. 权限点汇总与下发

- **后端任务**
  - 提供当前用户权限点汇总接口（两种方案，取其一即可）：
    - 方案 1：在 `/admin/auth/profile` 的 `data.buttons` 中附带所有权限点字符串。
    - 方案 2：新增 `GET /admin/auth/buttons`，专门返回权限点数组。
  - 汇总逻辑：
    - 根据用户 `roles` → `xy_admin_role_menu` → `xy_admin_menu.perms`，汇总去重。
    - 超级管理员直接返回所有菜单里的权限点。
- **前端任务**
  - 用户登录后，在用户 store 中保存 `buttons: string[]`。
  - 实现一个 `hasPerm(code: string)` 或 `v-perm` 指令，用于按钮级显示控制。
- **验收**
  - 某角色只配置了部分按钮的权限点：
    - 对应按钮显示正常，其他按钮不可见；
    - 前端强行调用接口时，如果没有权限点，后端直接拒绝（见下一小节）。

#### C-3. 接口权限中间件 `PermissionMiddleware`

- **后端任务**
  - 在 `middleware` 新增 `PermissionMiddleware`：
    - 挂载顺序：`CORS` → `AdminAuth` → `PermissionMiddleware` → 业务 Handler → `ResponseHandler`。
    - 功能：
      1. 从上下文获取当前用户权限点集合（可通过缓存/注入方式）。
      2. 根据当前请求构造权限字符串：`METHOD + PATH`，例如：`"GET /admin/user/list"`；路径建议使用路由模板。
      3. 如未命中权限点集合，返回错误码 `consts.CodeNoPermission`。
  - 可选优化：为高频接口加入权限点缓存（如 `map[roleKey][]perm`）。
- **前端任务**
  - 在 HTTP 封装中，根据 `CodeNoPermission` 做统一处理（如跳转 403 页或 toast 提示）。
- **验收**
  - 用户拥有菜单，但去掉某按钮的权限点：
    - 按钮在前端不显示；
    - 手动调用接口（如在 Apifox）会返回“无权限”错误码。

### 6. 阶段 D：数据权限（行级）

#### D-1. 角色数据范围编辑与接口

- **后端任务**
  - 在角色编辑接口 `/admin/role/save` 中支持 `data_scope/custom_depts` 字段保存。
  - 提供获取数据范围选项接口（可选）：`GET /admin/role/dataScopeOptions`（返回 ALL/DEPT/... 文案）。
- **前端任务**
  - 在角色编辑弹窗中增加“数据范围”配置：
    - 单选：全部 / 本部门 / 本部门及子部门 / 仅本人 / 自定义部门。
    - 当选择“自定义部门”时，弹出部门树多选框。
- **验收**
  - 角色保存后，DB 中 `data_scope/custom_depts` 字段正确更新，并可在编辑时回显。

#### D-2. 数据权限过滤实现（选一个列表先落地）

- **后端任务**
  - 选定一个列表接口作为样板（推荐：管理员列表 `/admin/user/list`）：
    - 在查询逻辑中引入数据权限过滤函数（如 `ApplyDataScope`）。
    - 根据当前用户的角色 `data_scope` 和部门关系，动态追加 `WHERE` 条件。
  - 封装通用工具（例如放在 `internal/service` 或 `internal/logic` 层）：
    - 输入：当前 `gdb.Model`、表别名、部门字段名、创建人字段名等。
    - 输出：带数据权限过滤的 `gdb.Model`。
- **前端任务**
  - 无需特别改动，仅用于验证效果（不同账号看到的列表内容不同）。
- **验收**
  - 建 2~3 个账号，配置不同角色和部门：
    - `data_scope=ALL` 的角色可见所有数据。
    - `DEPT` 角色只能看到本部门。
    - `SELF` 角色只能看到自己创建的数据。
