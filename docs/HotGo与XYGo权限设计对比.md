# HotGo 与 XYGo 权限设计对比分析

## 一、数据表结构对比

### 1.1 用户表

| 字段 | HotGo (`admin_member`) | XYGo (`admin_user`) | 差异说明 |
|------|----------------------|-------------------|---------|
| **主键** | id (int64) | id (uint64) | 类型不同 |
| **部门** | dept_id (int64) | dept_id (uint64) | ✅ 都有 |
| **角色** | **role_id** (int64) **单个** | ❌ 无直接字段 | **关键差异1** |
| **角色关联** | 1:1（直接字段） | N:M（中间表`admin_user_role`） | **设计理念不同** |
| **岗位** | N:M（`admin_member_post`） | N:M（`admin_user_post`） | ✅ 相同 |
| **上级** | pid (int64) | pid (uint64) | ✅ 都有 |
| **树结构** | level, tree | ❌ 无 | **差异2** |
| **真实姓名** | real_name | ❌ 无 | HotGo更详细 |
| **其他** | 积分、余额、生日、地址等 | ❌ 无 | HotGo字段更丰富 |

### 1.2 角色表

| 字段 | HotGo (`admin_role`) | XYGo (`admin_role`) | 差异说明 |

| **数据范围** | data_scope (int) | data_scope (int) | ✅ 相同 |

| **其他字段** | 基本一致 | 基本一致 | ✅ 设计相同 |

### 1.3 部门表

| 字段 | HotGo (`admin_dept`) | XYGo (`admin_dept`) | 差异说明 |
|------|---------------------|-------------------|---------|
| **树结构** | parent_id | parent_id + 闭包表 | XYGo多了闭包表 |
| **部门类型** | **type** (租户/商户/用户/部门) | ❌ 无 | **HotGo支持多租户** |
| **其他** | 基本字段相同 | 基本字段相同 | ✅ 核心一致 |

### 1.4 岗位表

| 字段 | HotGo (`admin_post`) | XYGo (`admin_post`) | 差异说明 |
|------|---------------------|-------------------|---------|
| **所有字段** | 完全一致 | 完全一致 | ✅ 设计相同 |

## 二、核心设计差异

### 2.1 用户-角色关系（最大差异）

#### HotGo 的设计

```
用户表（admin_member）
  └─ role_id (int64)  ← 直接存储单个角色ID

特点：
  - 一个用户只能有一个主角色
  - 简单直接，查询快
  - 不支持多角色
```

**优点**：
- 数据库设计简单
- 查询性能好（无需JOIN中间表）
- 符合大多数场景（一人一岗）

**缺点**：
- 不支持一个用户拥有多个角色
- 不灵活（如：用户既是技术主管又是项目经理）

#### XYGo 的设计

```
用户表（admin_user）
  └─ ❌ 无role_id字段

中间表（admin_user_role）
  ├─ user_id
  └─ role_id

特点：
  - 一个用户可以有多个角色
  - 多对多关系
  - 更灵活
```

**优点**：
- 支持多角色
- 灵活度高（一人多职）
- 权限取并集（用户拥有所有角色的权限）

**缺点**：
- 需要JOIN中间表（性能稍差）
- 数据库设计稍复杂

### 2.2 用户树结构

#### HotGo 的设计

```sql
admin_member:
  - pid: 上级ID
  - level: 层级
  - tree: 关系树路径（如 "0,1,3"）
```

**特点**：用户表直接维护树形关系

**用途**：
- 快速查询用户的所有上级
- 快速查询用户的所有下级
- 支持数据权限的"自己和下级"模式

#### XYGo 的设计

```sql
admin_user:
  - pid: 上级ID
  - ❌ 无 level, tree 字段
```

**特点**：只有上级ID，无预计算的树路径

**影响**：
- 查询下级需要递归查询
- `DataScope = 6,7`（自己和下级）性能稍差

### 2.3 多租户支持

#### HotGo 的设计

```sql
admin_dept:
  - type: 部门类型（dept=部门, tenant=租户, merchant=商户, user=用户）
```

**特点**：
- 部门表支持多种类型
- 可实现**SaaS多租户隔离**
- 三层隔离：租户 → 商户 → 用户

#### XYGo 的设计

```sql
admin_dept:
  - ❌ 无 type 字段
```

**特点**：
- 仅支持部门树
- 不支持多租户隔离
- 适用于**单租户企业内部系统**

## 五、详细差异分析

### 差异1：用户角色关系（最重要）

#### HotGo 设计哲学

```
一个用户 = 一个主角色 + 可选的岗位（辅助）
```

**理念**：角色是主要权限载体，岗位只是标签

**适用场景**：
- 组织架构清晰，职责明确
- 一人一岗制
- 角色划分细致（可以为每种职责创建角色）

#### XYGo 设计哲学

```
一个用户 = 多个角色 + 可选的岗位
```

**理念**：用户可以身兼多职，权限是所有角色的并集

**适用场景**：
- 小团队，一人多职
- 临时权限需求（不想创建新角色）
- 灵活的权限组合

### 差异2：用户树结构的性能优化

#### HotGo 的优化

```sql
admin_member:
  - pid: 1
  - level: 2
  - tree: "0,1,3"  ← 预计算的路径
```

**优势**：
- 查询所有下级：`WHERE tree LIKE '0,1,3,%'`（一条SQL搞定）
- 查询所有上级：解析 tree 字符串
- 适合频繁的树查询

**劣势**：
- 需要维护tree字段（插入/更新时计算）
- 数据迁移复杂

#### XYGo 的实现

```sql
admin_user:
  - pid: 1
  - ❌ 无 level, tree
```

**实现**：
- 查询所有下级：递归查询（`GetSelfAndAllSub` 函数）
- 实时计算，无需维护额外字段

**优势**：
- 数据一致性好
- 无需维护tree字段

**劣势**：
- 深层级查询性能差（需递归N次）
- 建议：如用户树超过3层，应添加tree字段优化

### 差异3：多租户支持

#### HotGo 的多租户设计

```
租户（type=tenant）
  └─ 商户（type=merchant）
      └─ 用户（type=user）
          └─ 普通部门（type=dept）
```

**场景**：SaaS平台，多个租户共用系统

**实现**：
- 部门表的 `type` 字段区分类型
- FilterTenant Handler 自动隔离租户数据
- 每个租户看不到其他租户的数据

#### XYGo 的设计

```
部门树（单一type）
  └─ 子部门
      └─ 子部门...
```

**场景**：单一企业内部系统

**特点**：
- 只支持部门隔离
- 不支持多租户
- 更简单，适合企业内部

## 七、优缺点总结

### HotGo 的优势

1. **性能优化**：
   - 用户树用 `tree` 字段预计算，查询快
   - 用户角色直接存储，无需JOIN

2. **多租户支持**：
   - 部门支持 `type` 字段区分租户
   - 适合SaaS平台

3. **成熟稳定**：
   - 经过多个项目验证
   - 文档和示例完善

### HotGo 的劣势

1. **一人一角色**：
   - 不支持用户多角色
   - 灵活性差

2. **tree字段维护**：
   - 需要额外逻辑维护树路径
   - 数据迁移复杂

### XYGo 的优势

1. **多角色支持**：
   - 用户可以拥有多个角色
   - 适合小团队，一人多职

2. **字段权限**：
   - 扩展了字段级权限控制
   - HotGo未实现的功能

3. **数据一致性**：
   - 用户树无冗余字段
   - 不需要维护tree字段

### XYGo 的劣势

1. **用户树性能**：
   - 深层级查询需递归
   - 大数据量下性能差

2. **无多租户**：
   - 不支持SaaS场景
   - 只适合单租户企业

3. **查询复杂度**：
   - 用户角色需JOIN中间表
   - SQL比HotGo复杂

## 八、适用场景建议

### HotGo 更适合

- ✅ **SaaS平台**：多租户、多商户
- ✅ **大型企业**：用户数>10000，组织层级深（>5层）
- ✅ **职责单一**：一人一岗，角色划分清晰
- ✅ **高并发场景**：需要极致的查询性能

### XYGo 更适合

- ✅ **单租户企业**：公司内部管理系统
- ✅ **中小团队**：用户数<5000，组织层级浅（<4层）
- ✅ **一人多职**：员工身兼多个角色
- ✅ **精细权限控制**：需要字段级权限

## 九、性能对比

### 查询用户角色

**HotGo**：
```sql
SELECT * FROM admin_member WHERE id = 100;
-- 一条SQL，role_id 直接在结果里
```

**XYGo**：
```sql
SELECT * FROM admin_user WHERE id = 100;
SELECT role_id FROM admin_user_role WHERE user_id = 100;  -- 需要额外查询
```

**性能差距**：HotGo 快 20-30%

### 查询用户所有下级

**HotGo**：
```sql
SELECT * FROM admin_member 
WHERE tree LIKE CONCAT((SELECT tree FROM admin_member WHERE id = 100), ',%');
-- 一条SQL，利用tree字段索引
```

**XYGo**：
```go
// 递归查询
func GetAllSubs(userId) []int {
    subs := []
    directSubs := SELECT id FROM admin_user WHERE pid = userId
    for _, sub := range directSubs {
        subs = append(subs, GetAllSubs(sub)...)  // 递归
    }
    return subs
}
```

**性能差距**：
- 3层以内：差距不大
- 5层以上：HotGo 快 5-10倍

## 十、迁移建议

### 如果要对齐HotGo，需要做什么？

#### 方案A：完全对齐（性能优先）

**修改**：
1. 用户表添加 `role_id` 字段（改为1:1）
2. 用户表添加 `level`, `tree` 字段
3. 删除 `admin_user_role` 中间表
4. 重写 FilterAuth Handler 的用户树查询部分

**代价**：
- 现有多角色用户需要选择主角色
- 需要迁移数据
- 前端用户管理页面需要调整

#### 方案B：混合设计（兼顾灵活性和性能）

**修改**：
1. **保留** N:M 的角色关系（支持多角色）
2. **添加** `level`, `tree` 字段到用户表（优化树查询）
3. **添加** `type` 字段到部门表（可选，为未来多租户预留）

**优势**：
- 保留XYGo的灵活性
- 提升树查询性能
- 为多租户预留扩展空间

#### 方案C：保持现状（适合当前场景）

**建议**：
- 如果用户数 < 5000，组织层级 < 4层
- **不需要修改**，当前设计已足够

**优化点**（可选）：
- 添加 `tree` 字段到用户表
- 在用户创建/更新时自动维护tree字段
- FilterAuth Handler 优先使用tree字段查询

## 十一、结论

### 设计理念对比

| 维度 | HotGo | XYGo |
|------|-------|------|
| **定位** | SaaS平台，大型企业 | 企业内部系统，中小团队 |
| **设计原则** | 性能优先，预计算 | 灵活优先，实时计算 |
| **复杂度** | 中等（tree维护） | 简单（无冗余字段） |
| **扩展性** | 高（多租户） | 中（单租户） |

### 核心差异总结

1. **用户角色关系**：
   - HotGo = 1:1（单角色）
   - XYGo = N:M（多角色）

2. **用户树优化**：
   - HotGo = tree字段（快）
   - XYGo = 递归查询（慢）

3. **多租户支持**：
   - HotGo = 支持（部门type）
   - XYGo = 不支持

4. **字段权限**：
   - HotGo = 无
   - XYGo = 有

### 最终建议

**当前XYGo的设计已经足够好**，除非有以下需求，否则**不建议大改**：

1. **用户数>5000** → 添加tree字段优化
2. **需要SaaS多租户** → 参考HotGo添加部门type
3. **强制一人一角色** → 改为HotGo的role_id设计

**XYGo的优势（字段权限+多角色）是差异化特性，应保留！**
