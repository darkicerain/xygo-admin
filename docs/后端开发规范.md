# 后端开发规范

## 一、GoFrame 三层架构

### 1.1 分层职责

```
Controller → Service → Logic → DAO
   ↓          ↓         ↓       ↓
 路由处理   接口定义  业务逻辑  数据访问
```

| 层级 | 路径 | 职责 | 生成方式 |
|------|------|------|----------|
| **Controller** | `internal/controller/admin/` | HTTP请求处理、参数验证、响应封装 | 手写 |
| **Service** | `internal/service/` | 接口定义（Interface） | **自动生成** |
| **Logic** | `internal/logic/` | 业务逻辑实现 | 手写 |
| **DAO** | `internal/dao/` | 数据库访问对象 | **自动生成** |

### 1.2 关键规则

#### Service 层（自动生成）
- ✅ **由 `gf gen service` 自动生成**
- ❌ **禁止手写修改**（每次生成会覆盖）
- 📝 文件头注释：`Code generated and maintained by GoFrame CLI tool. DO NOT EDIT.`

#### Logic 层（手写）
- ✅ **结构体命名**：必须以 `s` 开头（如 `sAdminUser`）
- ✅ **init函数**：注册到 Service
- ✅ **方法签名**：必须与 Service 接口一致

#### logic.go（自动生成）
- ✅ **由 `gf gen service` 自动生成**
- ✅ **自动扫描**：自动导入 `internal/logic/` 下所有子包
- ❌ **无需手动维护**：每次运行 `gf gen service` 会自动更新

## 二、标准开发流程

### 2.1 创建新模块（完整流程）

```bash
# 步骤1：创建数据表
mysql> CREATE TABLE xy_admin_post (...);

# 步骤2：生成 DAO 层
gf gen dao

# 步骤3：创建 Logic 层（手写）
# internal/logic/post/post.go
package post

type sAdminPost struct{}

func init() {
    service.RegisterAdminPost(New())
}

func New() *sAdminPost {
    return &sAdminPost{}
}

func (s *sAdminPost) List(ctx context.Context, in *adminin.PostListInp) ([]adminin.PostListItem, int, error) {
    // 业务逻辑
}

# 步骤4：注册 Logic 包
# internal/logic/logic.go
import (
    _ "xygo/internal/logic/post"  // ← 手动添加这一行
)

# 步骤5：生成 Service 接口
gf gen service

# 步骤5：创建 Controller（手写）
# internal/controller/admin/post.go
func (c *cAdminPost) List(ctx context.Context, req *admin.PostListReq) (res *admin.PostListRes, err error) {
    list, total, err := service.AdminPost().List(ctx, &adminin.PostListInp{...})
    // ...
}

# 步骤6：注册路由（在 internal/cmd/cmd.go）
group.POST("/post/list", api.AdminPost.List)
```

### 2.2 修改模块业务逻辑

```bash
# 步骤1：修改 Logic 层代码
vim internal/logic/user/user.go

# 步骤2：重新生成 Service 接口（如果方法签名改了）
gf gen service

# 步骤3：编译运行
go build
```

### 2.3 新增 Field 定义

```bash
# 步骤1：创建字段定义文件
# internal/field/admin_post.go
package field

var AdminPostFields = []ResourceField{
    {Code: "id", Name: "ID", Type: "uint64"},
    // ...
}

func init() {
    Register("admin_post", "岗位管理", AdminPostFields)
}

# 步骤2：无需其他操作（main.go已导入 _ "xygo/internal/field"）
```

## 三、`gf gen service` 工作原理

### 3.1 扫描规则

`gf gen service` 会：
1. 扫描 `internal/logic/` 目录
2. 查找符合规则的结构体：`^s([A-Z]\w+)$`（如 `sAdminUser`）
3. 提取该结构体的所有**公开方法**
4. 自动生成对应的 Service 接口文件

### 3.2 生成示例

**Logic 层代码**（手写）：
```go
// internal/logic/user/user.go
package user

type sAdminUser struct{}

func (s *sAdminUser) List(ctx context.Context, in *adminin.UserListInp) ([]adminin.UserListItem, int, error) {
    // ...
}

func (s *sAdminUser) Delete(ctx context.Context, id uint64) error {
    // ...
}
```

**执行 `gf gen service` 后自动生成**：
```go
// internal/service/user.go
// Code generated and maintained by GoFrame CLI tool. DO NOT EDIT.
package service

type IAdminUser interface {
    List(ctx context.Context, in *adminin.UserListInp) ([]adminin.UserListItem, int, error)
    Delete(ctx context.Context, id uint64) error
}

var localAdminUser IAdminUser

func AdminUser() IAdminUser {
    if localAdminUser == nil {
        panic("implement not found for interface IAdminUser, forgot register?")
    }
    return localAdminUser
}

func RegisterAdminUser(i IAdminUser) {
    localAdminUser = i
}
```

## 四、logic.go 管理方案

### 4.1 官方方案（自动生成）✅

**`gf gen service` 会自动生成 `logic.go`**：

```go
// internal/logic/logic.go
// ==========================================================================
// Code generated and maintained by GoFrame CLI tool. DO NOT EDIT.
// ==========================================================================

package logic

import (
    _ "xygo/internal/logic/dept"
    _ "xygo/internal/logic/post"
    _ "xygo/internal/logic/user"
    // ✅ 自动扫描 internal/logic/ 目录
)
```

**工作原理**：
1. `gf gen service` 扫描 `internal/logic/` 目录
2. 发现所有子目录（dept、post、user等）
3. 自动生成 `logic.go` 并导入所有子包
4. **无需手动维护**

### 4.2 辅助脚本（可选）

虽然 `gf gen service` 已经能自动生成 `logic.go`，但我们提供了 PowerShell 脚本 `gen.ps1` 用于快速开发：

```bash
# 单独生成某一层
.\gen.ps1 dao       # 生成 DAO 层
.\gen.ps1 service   # 生成 Service 层（包含 logic.go）
.\gen.ps1 logic     # 单独生成 logic.go（极少使用）

# 一键生成全部
.\gen.ps1 all       # dao + service（推荐）

# 编译运行
.\gen.ps1 build     # 编译
.\gen.ps1 run       # 热编译运行
```

## 五、常见问题

### Q1：为什么 Service 文件会被覆盖？
**A**：Service 层是**自动生成**的，每次运行 `gf gen service` 都会重新生成。
- ✅ 业务逻辑应写在 **Logic 层**
- ❌ 不要在 Service 文件中添加代码

### Q2：如何添加新的 Service 方法？
**A**：在 Logic 层添加方法，然后运行 `gf gen service`：
```go
// 1. 在 logic/user/user.go 添加方法
func (s *sAdminUser) ResetPassword(ctx context.Context, id uint64) error {
    // ...
}

// 2. 运行命令
gf gen service

// 3. Service 接口会自动更新
```

### Q3：新增模块是否必须手动修改 logic.go？
**A**：**不需要**！`gf gen service` 会自动扫描 `internal/logic/` 目录并生成 `logic.go`。

### Q4：logic.go 可以自动生成吗？
**A**：**可以**！运行 `gf gen service` 或 `.\gen.ps1 all` 即可自动生成。

## 七、核心原则

1. **Service 接口自动生成，Logic 实现手写**
2. **DAO 层完全自动生成，不要手动修改**
3. **Controller 层手写，调用 Service**
4. **logic.go 手动维护（或自动化脚本生成）**
5. **所有业务逻辑必须在 Logic 层**
6. **数据库表改动后立即运行 `gf gen dao`**
7. **Logic 方法签名改动后立即运行 `gf gen service`**
