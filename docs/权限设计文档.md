### 权限设计文档（参考 HotGo + 结合本项目约定）

> 目的：梳理一套**可复用、可扩展**的权限模型，既借鉴 HotGo 成熟做法，又符合本项目已拍板的规范（BuildAdmin 风格 RBAC、`METHOD + PATH` 权限点、部门闭包表等）。  
> 范围：用户 / 角色 / 菜单 / 按钮权限 / 数据权限（行级）/ 鉴权流程。

---

### 1. HotGo 权限模型拆解（我们要学的部分）

#### 1.1 核心表结构（后端）

- **用户表 `admin_member`**（此处只列出与权限直接相关字段）：
  - `id`：用户 ID。
  - `dept_id`：所属部门。
  - 其它：用户名、密码、状态等。
- **角色表 `admin_role`**（树形结构）：
  - `id`：角色 ID。
  - `name`：角色名称（显示用）。
  - `key`：角色编码（权限标识，如 `R_SUPER`）。
  - `data_scope`：数据范围（全部、本部门、本部门及子部门、仅本人、自定义部门）。
  - `custom_dept`：自定义部门 JSON（`[]int64`）。
  - `pid`：上级角色 ID。
  - `level`：树层级。
  - `tree`：树路径（形如 `"0,1,3"`）。
  - `status`：状态（启用/禁用）。
- **菜单表 `admin_menu`**（树形 + 菜单/按钮）：
  - `id/pid/level/tree`：树结构。
  - `title`：菜单名称。
  - `name`：路由名称编码。
  - `path`：路由路径。
  - `type`：菜单类型（`1目录 2菜单 3按钮`）。
  - `permissions`：菜单包含权限集合（字符串，内部再拆）。
  - `permission_name`：权限名称。
  - `component`/`icon`/`hidden`/`affix` 等前端路由元信息。
  - `status`：启用/禁用。
- **关联表：**
  - `admin_member_role`：用户–角色多对多绑定。
  - `admin_role_menu`：角色–菜单/按钮多对多绑定（决定角色能看到哪些菜单/按钮）。
  - `admin_role_casbin`：角色–Casbin 策略表（HotGo 用它做接口级鉴权，我们不会直接用 Casbin，但会借鉴其“角色→权限点列表”的思想）。
- **部门表 `admin_dept`**：
  - 采用与角色类似的树结构（`pid/level/tree`），用于数据权限中的“部门及子部门”等规则。

#### 1.2 HotGo 的权限维度

1. **菜单权限（可见性）：**
   - 通过 `admin_role_menu` 决定某个角色能看到哪些菜单/按钮。
   - 菜单为树，按钮挂在菜单下面，按钮本身也有 `permissions` 字段。
2. **接口权限（调用能力）：**
   - HotGo 使用 Casbin：把菜单/按钮里配置的权限点同步到 `admin_role_casbin`，按 `sub=role.key, obj=resource, act=method` 匹配。
3. **数据权限（行级）：**
   - 在角色上配置 `data_scope` + `custom_dept`。
   - 在查询时，根据当前用户角色/部门信息构造 `WHERE` 条件（本部门、本部门及子部门、本人、自定义部门集合）。
4. **多维度结合：**
   - **菜单控制“看得见”**：不在角色菜单树里的页面和按钮，前端不展示。
   - **接口控制“调得通”**：即便前端改代码强行发请求，如果没有相应权限点，后端直接拒绝。
   - **数据权限控制“看多少”**：在能调通的接口里，行级过滤保证“最多看到自己该看的数据”。

#### 1.3 角色树在 HotGo 中的作用

- `admin_role` 之所以做成树，主要用于：
  - **展示分组**：例如“租户 → 商家 → 用户”，“管理 → 业务线角色”等，界面上更直观。
  - **批量继承/复制策略**（可作为业务语义）：
    - 子角色可在 UI 上“继承/复制”父角色的菜单、数据范围，然后再做微调。
- 注意：HotGo 实际的权限计算仍然基于“**当前用户直接绑定的角色集合**”，而不是自动沿树向上继承所有父角色；树更偏向于**组织和管理**角色，而不是复杂的多级继承。

### 2. 本项目权限目标（在 HotGo 之上做的取舍）

#### 2.1 已拍板的前置决策

- **RBAC 风格**：采用 BuildAdmin 风格的**传统 RBAC（三表）**，而不是 Casbin。
  - 用户–角色：`xy_admin_user_role`。
  - 角色–菜单：`xy_admin_role_menu`。
  - 权限点：挂在菜单/按钮上。
- **权限点格式**：统一为 `METHOD + PATH`，例如：
  - `GET /admin/user/list`
  - `POST /admin/role/save`
- **主键策略**：整型自增 `INT UNSIGNED`。
- **部门树**：采用**闭包表**（`xy_admin_dept` + `xy_admin_dept_closure`）。
- **API 返回结构**：统一 `{ code, message, data, traceId }`。
- **错误码**：集中在 `internal/consts/code.go`，业务代码只能使用 `consts.CodeXXX`。

#### 2.2 我们采用的 HotGo 设计点

- 借鉴：
  - 角色树（`pid/level/tree`）用于**角色分组和可视化管理**。
  - 菜单树（目录/菜单/按钮三类）+ 丰富的路由 `meta` 字段。
  - 角色 `data_scope + custom_depts` 的数据权限模型。
  - 角色–菜单绑定 + 菜单上挂权限点的模式。
- 不采用：
  - Casbin 引擎和 `admin_role_casbin` 表；我们用自研中间件按 `METHOD + PATH` 做匹配即可。

### 3. 我们的数据模型设计（xygo 实际落地）

#### 3.1 核心表对照

- **用户表 `xy_admin_user`**（已建表）：
  - 关键字段：`id/username/password/salt/nickname/is_super/dept_id/status/...`
  - 用于登录、菜单权限、数据权限的主体。

- **角色表 `xy_admin_role`**（已建表）：
  - `id`：角色 ID。
  - `name`：角色名称（展示）。
  - `key`：角色编码（例如 `R_SUPER/R_ADMIN/R_USER`，与前端路由 meta.roles 一一对应）。
  - `data_scope`：数据范围（对齐 HotGo）
    - `0`：全部数据。
    - `1`：本部门。
    - `2`：本部门及子部门。
    - `3`：仅本人。
    - `4`：自定义部门集合。
  - `custom_depts`：JSON 数组，存自定义部门 ID 列表。
  - `pid/level/tree`：角色树结构（**用于展示/分组/复制，不做自动继承权限**）。
  - `remark/sort/status/created_at/...`：说明、排序、状态与审计字段。

- **角色关联表：**
  - `xy_admin_user_role(user_id, role_id)`：用户–角色多对多。
  - `xy_admin_role_menu(role_id, menu_id)`：角色–菜单/按钮多对多。

- **菜单表 `xy_admin_menu`**（已建表）：
  - `id/pid/level/tree`：菜单树结构。
  - `title`：菜单名称（i18n key 可后置）。
  - `name`：前端路由名。
  - `path`：路由路径。
  - `type`：菜单类型（1目录 / 2菜单 / 3按钮）。
  - `component/icon/redirect/hidden/affix/keep_alive/...`：前端路由元信息。
  - `perms`（建议字段名）：权限点数组 JSON，元素为 `METHOD + PATH` 字符串。
    - 目录/菜单通常为空或只配少量权限点。
    - 按钮型菜单重点配置接口权限点。
  - `status/sort/remark/created_at/...`。

- **部门表 `xy_admin_dept` + 闭包表 `xy_admin_dept_closure`**：
  - 用于实现 `data_scope` 中“部门及子部门”、“自定义部门集合”等规则。

#### 3.2 角色树的语义（在 xygo 中的具体定义）

- **树仅用于“组织角色”和 UI 展示，不做自动继承权限**：
  - 父角色不会自动把自己的菜单/数据范围下发给所有子角色。
  - 业务上如需“复制父角色权限”，通过“复制角色/一键继承”接口显式实现。
- UI 层面的行为：
  - 角色管理页面按 `pid/tree` 展示为树形列表（类似 HotGo 与截图中的效果）。
  - “添加子角色”时，自动填充 `pid/level/tree` 字段，并可选择是否复制父角色菜单绑定与数据范围。

### 4. 权限判定流程（请求生命周期）

#### 4.1 登录与会话

1. 用户调用 `POST /admin/auth/login`，后端：
   - 校验用户名/密码。
   - 写入会话（当前实现为文件存储，后续换 Redis）。
   - 生成 `accessToken`，返回给前端。
2. 前端将 `accessToken` 放入 `Authorization` 头（目前格式为纯 token，后续可统一为 `Bearer xxx`）。

#### 4.2 鉴权中间件链

- `AdminAuth`（已实现）：
  - 放行 `/admin/auth/login`。
  - 其它 `/admin/**`：解析 `Authorization`，调用 `token.Parse` 校验签名、过期、会话存在性。
  - 失败时设置错误码 `consts.CodeNotAuthorized`（当前前端用 `61` 映射）。
- **后续将新增 `PermissionMiddleware`**：
  - 在 `AdminAuth` 之后执行。
  - 责任：根据当前用户 `roles` + 角色–菜单–权限点关系，判断当前 `METHOD + PATH` 是否在权限集合内。

#### 4.3 菜单 & 接口权限流程

1. **加载用户信息与角色列表**：
   - `/admin/auth/profile` 返回：
     - `id/username/nickname/isSuper`。
     - `roles: []string`：来自 `xy_admin_role.key`，通过 `xy_admin_user_role` 计算。
     - `buttons: []string`：预留的按钮权限点列表（后续会从菜单权限点中汇总）。
2. **加载菜单树**（规划接口示例）：
   - `GET /admin/menu/routes`：返回当前用户可见的菜单/按钮树：
     - 根据 `roles` → `xy_admin_role_menu` → `xy_admin_menu` 过滤。
     - 只返回 `status=1` 的菜单。
     - 节点包含路由需要的 `path/name/component/meta` 等。
3. **按钮权限**：
   - 菜单节点里的 `perms` 字段统一使用 `METHOD + PATH` 字符串数组。
   - 登录后，后端可以：
     - 直接在 `/admin/auth/profile` 中附带一个合并后的 `buttons: []string`；
     - 或提供单独接口 `GET /admin/auth/buttons` 返回当前用户所有权限点。
   - 前端按钮显示：基于 `buttons` 做 `includes` 判断。
4. **接口鉴权（PermissionMiddleware）**：
   - 组合当前请求的 `METHOD + PATH` 作为权限点字符串（注意 path 使用路由模板形式，例如 `/admin/user/detail/:id`）。
   - 判断该字符串是否在当前用户权限集合中：
     - 超级管理员（`isSuper = true` 或拥有特殊角色，如 `R_SUPER`）可直接放行。
     - 否则，若权限点不存在，则返回 `consts.CodeNoPermission`。

### 5. 数据权限（行级）设计

#### 5.1 数据范围字段（对齐 HotGo）

- `xy_admin_role.data_scope`：
  - `0`：全部数据 (`ALL`)。
  - `1`：本部门 (`DEPT`)。
  - `2`：本部门及子部门 (`DEPT_AND_CHILD`)。
  - `3`：仅本人 (`SELF`)。
  - `4`：自定义部门集合 (`CUSTOM`)。
- `xy_admin_role.custom_depts`：存放自定义部门 ID 列表（JSON 数组）。

#### 5.2 实现策略

1. **构造“可见部门集合”**：
   - 基于当前用户的部门 ID 和其绑定角色的 `data_scope`，使用闭包表 `xy_admin_dept_closure` 得到一组部门 ID：
     - `DEPT`：当前 `dept_id`。
     - `DEPT_AND_CHILD`：`SELECT descendant_id FROM xy_admin_dept_closure WHERE ancestor_id = :dept_id`。
     - `CUSTOM`：汇总 `custom_depts`。
2. **统一过滤函数（推荐在 Service 层实现）**：
   - `ApplyDataScope(ctx, model, tableAlias, deptField, ownerField)`：
     - 根据当前登录用户上下文，自动在 SQL 中追加：
       - `deptField IN (:visibleDeptIds)` 或
       - `ownerField = :userId`（SELF）。
3. **适用范围：**
   - 列表/导出接口：全部走数据权限过滤。
   - 详情/更新/删除接口：在操作前调用 `AssertCanAccess(ctx, id)`，校验该行记录是否满足数据权限。

### 6. 前后端契约小结

- **角色相关：**
  - 表：`xy_admin_role` / `xy_admin_user_role` / `xy_admin_role_menu`。
  - 前端：
    - 角色管理页使用树形展示（基于 `pid/tree`）。
    - 路由 meta 中的 `roles: []` 与 `xy_admin_role.key` 对齐。
  - 后端：
    - `/admin/auth/profile` 返回 `roles`。
    - 角色仅作为**权限集合标签**，不做自动继承。

- **菜单 & 按钮：**
  - 表：`xy_admin_menu`（type=1/2/3）。
  - 权限点字段：`perms`（JSON 数组，元素 `METHOD + PATH`）。
  - 接口：
    - `GET /admin/menu/tree`：角色菜单树管理。
    - `GET /admin/menu/routes`：当前用户路由菜单树。

- **接口鉴权：**
  - 登录：`/admin/auth/login` → 返回 `accessToken`。
  - 用户信息/角色：`/admin/auth/profile` → `roles/buttons`。
  - 鉴权中间件：`AdminAuth` + `PermissionMiddleware`。
  - 统一返回：`{ code, message, data, traceId }`，未登录/无权限/数据权限不足分别有独立错误码。
