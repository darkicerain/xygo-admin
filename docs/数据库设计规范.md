# 数据库设计规范

> 本规范适用于 XYGo Admin 框架。遵循此规范设计数据表，搭配**可视化 CRUD 代码生成器**，可自动推导出正确的表单组件、搜索方式和前端页面，同时确保 **MySQL 与 PostgreSQL 双库兼容**。

---

## 一、表名与主键

| 规则 | 说明 |
|------|------|
| **表前缀** | 统一使用 `xy_`（在 `config.yaml` 中配置 `removePrefix`，生成代码时自动去除） |
| **命名风格** | `snake_case`，如 `xy_admin_user`、`xy_member_score_log` |
| **主键** | 每张表**必须有单字段自增主键**，不支持复合主键 |
| **主键名称** | 推荐统一使用 `id` |
| **主键类型** | MySQL: `bigint(20) UNSIGNED NOT NULL AUTO_INCREMENT`<br>PG: `bigint NOT NULL`（配合 SEQUENCE 实现自增） |

**表注释** 将被解析为菜单名称（自动去除"表"字并追加"管理"），例如：
- `会员表` → 菜单名：`会员管理`
- `订单流水表` → 菜单名：`订单流水管理`

## 二、字段类型对照表（MySQL ↔ PostgreSQL）

设计数据表时请以 MySQL 类型为基准，框架会在建表和代码生成时自动转换为 PG 等价类型。

### 2.1 整数类型

| MySQL 类型 | PG 等价类型 | Go 类型 (gf gen dao) | 说明 |
|-----------|-----------|---------------------|------|
| `tinyint(1)` | `smallint` | `int` | 状态/开关字段，**不要用 boolean** |
| `tinyint(4)` / `tinyint unsigned` | `smallint` | `int` | 小范围枚举值 |
| `int(10) unsigned` | `integer` | MySQL: `uint` / PG: `int` | 普通整数（见下方兼容说明） |
| `int(11)` | `integer` | `int` | 有符号整数（sort/score 等） |
| `bigint(20) unsigned` | `bigint` | `uint64` | ID、时间戳（通过 typeMapping 统一） |
| `bigint(20)` | `bigint` | MySQL: `int64` / PG: `uint64` | 有符号大整数（极少使用） |

> **兼容重点：PG 没有 unsigned 类型**
>
> - `bigint` 通过 `hack/config.yaml` 的 `typeMapping: int8 → uint64` 统一，两边都生成 `uint64`
> - `int unsigned` 无法通过 typeMapping 统一（会影响有符号字段），差异通过 logic 层 `uint()` 显式转换兼容
> - **设计建议**：时间戳和 ID 使用 `bigint`；普通计数/排序使用 `int`（有符号）

### 2.2 浮点 / 精确数值

| MySQL 类型 | PG 等价类型 | Go 类型 | 说明 |
|-----------|-----------|---------|------|
| `decimal(M,N)` | `numeric(M,N)` | `float64` | 金额等精确数值 |
| `float` | `real` | `float32` | 单精度浮点 |
| `double` | `double precision` | `float64` | 双精度浮点 |

### 2.3 字符串

| MySQL 类型 | PG 等价类型 | Go 类型 | 说明 |
|-----------|-----------|---------|------|
| `varchar(N)` | `varchar(N)` / `character varying(N)` | `string` | 变长字符串 |
| `char(N)` | `char(N)` | `string` | 定长字符串 |
| `text` | `text` | `string` | 长文本 |
| `longtext` / `mediumtext` | `text` | `string` | PG 只有 `text`，无长度限制 |

### 2.4 时间

| MySQL 类型 | PG 等价类型 | Go 类型 | 说明 |
|-----------|-----------|---------|------|
| `datetime` / `timestamp` | `timestamp` | `*gtime.Time` | 日期时间 |
| `date` | `date` | `*gtime.Time` | 日期 |
| `time` | `time` | `string` | 时间 |

> **推荐**：本框架使用 **bigint 存储 Unix 时间戳**（而非 datetime 类型），字段命名以 `_at` 或 `_time` 结尾，框架自动识别为时间戳字段。

### 2.5 其他

| MySQL 类型 | PG 等价类型 | Go 类型 | 说明 |
|-----------|-----------|---------|------|
| `json` | `jsonb` | `*gjson.Json` | PG 推荐用 `jsonb`（可索引） |
| `enum` | `varchar(50)` | `string` | PG 无原生 enum，转为 varchar |
| `blob` / 二进制 | `bytea` | `[]byte` | 二进制数据 |

### 2.6 禁止使用的类型

| 类型 | 原因 | 替代方案 |
|------|------|---------|
| `boolean` / `bool` | PG 的 boolean 与 MySQL 的 tinyint 生成不同 Go 类型，导致代码不兼容 | 使用 `smallint`（PG）/ `tinyint(1)`（MySQL） |
| `set` | PG 不支持 | 使用 `varchar` + JSON 数组 |
| `enum` | PG 不原生支持，且不利于维护 | 使用 `tinyint` / `smallint` + 注释字典 |

## 三、字段命名规范

### 3.1 通用命名

| 字段名 | 类型 | 说明 | 自动行为 |
|--------|------|------|---------|
| `id` | `bigint unsigned` | 主键 | 自动识别为主键，不出现在表单中 |
| `created_at` | `bigint unsigned` | 创建时间（Unix 时间戳） | 自动维护，不出现在表单中 |
| `updated_at` | `bigint unsigned` | 更新时间（Unix 时间戳） | 自动维护，不出现在表单中 |
| `deleted_at` | `bigint unsigned DEFAULT 0` | 软删除标记 | 自动维护，0=未删除，>0=删除时间戳 |
| `create_time` | `int unsigned` | 创建时间（备选命名） | 同 created_at |
| `update_time` | `int unsigned` | 更新时间（备选命名） | 同 updated_at |
| `created_by` | `bigint unsigned` | 创建人ID | 自动维护 |
| `updated_by` | `bigint unsigned` | 更新人ID | 自动维护 |
| `sort` | `int` | 排序权重 | 自动识别为排序字段 |
| `status` | `tinyint(1)` | 状态字段 | 自动生成单选框（配合注释字典） |

> **关于 `deleted_at` 的兼容要求**：
>
> - MySQL: `bigint(20) UNSIGNED NOT NULL DEFAULT '0'`
> - PG: `bigint NOT NULL DEFAULT 0`
> - **必须 NOT NULL DEFAULT 0**，不能允许 NULL。GoFrame 软删除条件是 `deleted_at = 0`，如果值为 NULL 则 `NULL = 0` 为 false，会导致查不到数据。

以下字段名会自动从列表/表单中隐藏：

`deleted_at`、`delete_time`、`options`、`content`、`body`、`password`、`salt`、`secret`

## 五、字段注释规范

字段注释是代码生成器**推导字典选项**的重要依据。

### 5.1 基本格式

```
标签描述:值1=选项1,值2=选项2
```

**示例：**

```sql
-- MySQL
`status` tinyint(1) NOT NULL DEFAULT '1' COMMENT '状态:0=禁用,1=启用'
`type` tinyint(1) NOT NULL DEFAULT '1' COMMENT '类型:1=通知,2=公告,3=私信'
`gender` tinyint(1) NOT NULL DEFAULT '0' COMMENT '性别:0=未知,1=男,2=女'

-- PG（等价）
status smallint NOT NULL DEFAULT 1   -- COMMENT ON COLUMN ... IS '状态:0=禁用,1=启用'
type smallint NOT NULL DEFAULT 1     -- COMMENT ON COLUMN ... IS '类型:1=通知,2=公告,3=私信'
gender smallint NOT NULL DEFAULT 0   -- COMMENT ON COLUMN ... IS '性别:0=未知,1=男,2=女'
```

### 5.2 注释解析规则

注释中同时包含 `:` (或 `：`) 和 `=` 时，会被解析为**选项字典**，自动生成单选框/标签渲染。

| 注释格式 | 自动生成 | 前端效果 |
|---------|---------|---------|
| `状态:0=禁用,1=启用` | radio 单选 + tag 标签 | 列表中显示彩色标签 |
| `类型:1=通知,2=公告,3=私信` | radio 单选 + tag 标签 | 列表中显示彩色标签 |
| `普通备注` | 无字典 | 普通文本显示 |

### 5.3 注释最佳实践

- **有选项的字段**：务必使用 `描述:值=标签` 格式，如 `状态:0=禁用,1=启用`
- **无选项的字段**：直接写描述，如 `用户名`、`手机号`
- **注释会成为**：Go 结构体注释、前端表单 label、API 文档描述

## 六、PG 兼容注意事项

### 6.1 不要使用 boolean

**错误写法（PG）：**
```sql
is_muted boolean DEFAULT false NOT NULL
```

**正确写法（PG）：**
```sql
is_muted smallint DEFAULT 0 NOT NULL
```

原因：`boolean` 在 `gf gen dao` 中可能映射为 Go `bool`，与 MySQL `tinyint` → `int` 不一致，导致业务代码类型不匹配。

### 6.2 不要使用 json，使用 jsonb

```sql
-- 错误
options json

-- 正确
options jsonb
```

PG 的 `jsonb` 支持索引和更高效的查询，且框架 `mysqlTypeToPgType` 已将 MySQL `json` 映射为 `jsonb`。

### 6.3 默认值写法差异

| 场景 | MySQL | PG |
|------|-------|-----|
| 整数默认值 | `DEFAULT '0'` 或 `DEFAULT 0` | `DEFAULT 0` |
| 字符串默认值 | `DEFAULT ''` | `DEFAULT ''::character varying` |
| 主键自增 | `AUTO_INCREMENT` | `SEQUENCE` + `nextval()` |

### 6.4 deleted_at 必须 NOT NULL DEFAULT 0

```sql
-- MySQL
`deleted_at` bigint(20) UNSIGNED NOT NULL DEFAULT '0' COMMENT 'deleted time'

-- PG
deleted_at bigint NOT NULL DEFAULT 0
```

**绝对不能**用 `DEFAULT NULL` 或允许 NULL，否则 GoFrame 的软删除条件 `deleted_at = 0` 会因 `NULL ≠ 0` 而过滤掉所有记录。

### 6.5 LEFT JOIN 与软删除

当 LEFT JOIN 关联了带 `deleted_at` 的表时，代码生成器会自动：
1. 添加 `.Unscoped()` 禁止框架在 WHERE 中注入 `deleted_at = 0`
2. 将 `deleted_at = 0` 放入 ON 条件中

这确保了 LEFT JOIN 的正常行为——关联不到时返回 NULL 而不是被过滤掉。

### 6.6 标识符引号

- MySQL 用反引号：`` `column_name` ``
- PG 用双引号：`"column_name"`
- 框架的 `dbdialect.QuoteIdentifier()` 自动处理，**业务代码中不要硬编码引号**

## 七、推荐的表结构模板

### MySQL

```sql
CREATE TABLE `xy_example` (
  `id`          bigint(20) UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  `name`        varchar(100)        NOT NULL DEFAULT '' COMMENT '名称',
  `category_id` bigint(20) UNSIGNED NOT NULL DEFAULT '0' COMMENT '分类ID',
  `cover_image` varchar(500)        NOT NULL DEFAULT '' COMMENT '封面图',
  `content`     text                     NULL COMMENT '内容',
  `amount`      decimal(10,2)       NOT NULL DEFAULT '0.00' COMMENT '金额',
  `sort`        int(11)             NOT NULL DEFAULT '0' COMMENT '排序',
  `status`      tinyint(1)          NOT NULL DEFAULT '1' COMMENT '状态:0=禁用,1=启用',
  `remark`      varchar(255)        NOT NULL DEFAULT '' COMMENT '备注',
  `created_at`  bigint(20) UNSIGNED      NULL COMMENT '创建时间',
  `updated_at`  bigint(20) UNSIGNED      NULL COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_category_id` (`category_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='示例表';
```

### PostgreSQL（等价）

```sql
CREATE TABLE public.xy_example (
    id          bigint NOT NULL,
    name        character varying(100) DEFAULT ''::character varying NOT NULL,
    category_id bigint DEFAULT 0 NOT NULL,
    cover_image character varying(500) DEFAULT ''::character varying NOT NULL,
    content     text,
    amount      numeric(10,2) DEFAULT 0.00 NOT NULL,
    sort        integer DEFAULT 0 NOT NULL,
    status      smallint DEFAULT 1 NOT NULL,
    remark      character varying(255) DEFAULT ''::character varying NOT NULL,
    created_at  bigint,
    updated_at  bigint
);

-- 注释
COMMENT ON TABLE public.xy_example IS '示例表';
COMMENT ON COLUMN public.xy_example.name IS '名称';
COMMENT ON COLUMN public.xy_example.category_id IS '分类ID';
COMMENT ON COLUMN public.xy_example.cover_image IS '封面图';
COMMENT ON COLUMN public.xy_example.content IS '内容';
COMMENT ON COLUMN public.xy_example.amount IS '金额';
COMMENT ON COLUMN public.xy_example.sort IS '排序';
COMMENT ON COLUMN public.xy_example.status IS '状态:0=禁用,1=启用';
COMMENT ON COLUMN public.xy_example.remark IS '备注';
COMMENT ON COLUMN public.xy_example.created_at IS '创建时间';
COMMENT ON COLUMN public.xy_example.updated_at IS '更新时间';

-- 自增序列
CREATE SEQUENCE public.xy_example_id_seq START WITH 1 INCREMENT BY 1 NO MINVALUE NO MAXVALUE CACHE 1;
ALTER SEQUENCE public.xy_example_id_seq OWNED BY public.xy_example.id;
ALTER TABLE ONLY public.xy_example ALTER COLUMN id SET DEFAULT nextval('public.xy_example_id_seq'::regclass);

-- 主键 & 索引
ALTER TABLE ONLY public.xy_example ADD CONSTRAINT xy_example_pkey PRIMARY KEY (id);
CREATE INDEX idx_xy_example_category_id ON public.xy_example (category_id);
```

## 八、hack/config.yaml 配置说明

```yaml
gfcli:
  gen:
    dao:
    # 切换数据库：二选一，启用一行、注释另一行（typeMapping 两边通用不用动）
    # - link: "mysql:user:pass@tcp(127.0.0.1:3306)/dbname?loc=Local&parseTime=true"  # MySQL
      - link: "pgsql:user:pass@tcp(127.0.0.1:5432)/dbname"                           # PG（当前）
        removePrefix: "xy_"
        descriptionTag: true
        jsonCase: "CamelLower"
        gJsonSupport: true
        clear: true
        # typeMapping 两边通用，无需随数据库切换修改：
        # - PG bigint(int8) 默认生成 int64，映射为 uint64 与 MySQL bigint unsigned 对齐
        # - MySQL bigint unsigned 本身就是 uint64，此映射无副作用
        typeMapping:
          int8:
            type: uint64
```

## 十、常见问题

### Q: 为什么 PG 上查不到数据？
检查是否有 LEFT JOIN 关联了带 `deleted_at` 的表。GoFrame 会自动在 WHERE 中注入 `deleted_at = 0`，将 LEFT JOIN 变成 INNER JOIN 效果。解决方案：使用 `.Unscoped()` + 在 ON 条件中加 `deleted_at = 0`。代码生成器已自动处理。

### Q: 为什么 PG 上 status 字段报错 "smallint = boolean"？
说明你的 PG 表里 status 还是 `boolean` 类型。请改为 `smallint`。业务代码中不要写 `status = false`，统一用 `status = 0`。

### Q: 切换 MySQL/PG 后需要重新 gen dao 吗？
需要。切换 `hack/config.yaml` 中的 link 后执行 `gf gen dao`，Entity 类型会根据目标库重新生成。`typeMapping` 不用改。

### Q: int unsigned 在 PG 上类型不一致怎么办？
在 logic 层赋值时用 `uint()` 显式转换。例如：`CreatedAt: uint(item.CreatedAt)`。这样两边 Entity 不同类型都能编译通过。
