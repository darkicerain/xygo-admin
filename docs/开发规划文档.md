### 项目开发规划文档（按天执行版）

> 目标：基于前端 `art-design-pro`（`https://gitee.com/lingchen163/art-design-pro`） + 后端 `GoFrame`（`https://goframe.org/`）打造一套类似 `HotGo`（`https://gitee.com/bufanyun/hotgo`）的可复用中后台管理框架。  
> 工作节奏：**每天 5 小时**，以“可验收的每日交付”为原则推进。

---

### 0. 总体目标与边界

#### 0.1 必须具备（MVP）
- **前后端分离**：后端提供 REST API；前端基于 `art-design-pro` 渲染管理后台。
- **数据库兼容**：**MySQL 8 + PostgreSQL** 同一套业务代码可运行（允许少量方言适配层，但不能“写两套 SQL”）。
- **认证鉴权**：JWT + **单点登录（SSO）** + **踢人**（强制下线）+ Token 刷新。
- **权限体系**：
  - **菜单权限**：后端返回动态菜单/路由树（前端动态注入路由）。
  - **按钮权限**：前端控制按钮显示/可用；后端接口强校验权限点。
  - **数据权限（行级）**：支持按**部门/部门及子部门/本人（创建人）/自定义**等数据范围；对列表与导出等接口生效。
- **多语言**：中英双语（i18n），支持动态切换并持久化。
- **主题**：暗黑模式 + 动态主题（至少主色/布局密度等），支持持久化。
- **基础系统模块**：用户/角色/菜单/部门/岗位/字典/参数配置/审计日志/文件上传（可选OSS后置）等。
- **规范先行**：接口返回、错误码、权限标识、审计字段、数据库命名、前端路由/菜单元信息等必须先统一。

#### 0.2 明确后置（后期做）
- **代码生成器**：放在基础规范与核心能力稳定后（本规划第 33～40 天）。
- **多租户**：本期不做（如未来需要，会在数据权限与表设计阶段预留扩展点）。
- **微服务**：本期单体优先（后续可拆）。

### 1. 推荐仓库结构（单仓 Monorepo）

#### 1.1 顶层目录
- `server/`：GoFrame 后端（建议采用 HotGo 风格：`cmd` 多入口 + `router` 按应用分组）
- `web/`：art-design-pro 前端（建议 `web` 命名，便于未来对齐“生成器输出路径”）
- `deploy/`：Docker、Nginx、脚本
- `docs/`：规范、设计、开发指南（本文在此）

#### 1.2 后端推荐分层（GoFrame）
- `server/internal/controller/`：接口层（HTTP）
- `server/internal/service/`：业务层接口（可测试/可替换）
- `server/internal/dao/`：数据访问层（Repo/DAO）
- `server/internal/model/`：DTO/VO/实体/枚举
- `server/internal/middleware/`：认证、权限、审计、限流等
- `server/internal/logic/`：业务实现层（通过 `init()` 注入到 `service`）
- `server/internal/router/`：路由注册（按 admin/api/home/websocket 等入口分组）
- `server/internal/cmd/`：命令行入口（推荐 `http/cron/queue/tools` 多入口）
- `server/manifest/config/`：配置文件（分环境）
- `server/resource/`：模板、字典初始化等（可选）

#### 1.3 前端推荐结构（art-design-pro 内）
- `web/src/services/`：请求封装与 API 定义
- `web/src/access/`：权限控制（按钮级）
- `web/src/locales/`：中英语言包
- `web/src/theme/`：主题/暗黑模式
- `web/src/pages/System/`：系统管理模块
- `web/src/pages/Monitor/`：日志监控模块
- `web/src/utils/`：工具方法（token、i18n、storage）

### 2. 核心规范（必须先定）

#### 2.1 API 返回规范（统一结构）
- **成功**：
  - `code`: 0
  - `message`: "ok"
  - `data`: any
  - `traceId`: string（可选）
- **失败**：
  - `code`: 业务错误码（非 0）
  - `message`: 可读错误信息（可 i18n 错误消息后置）
  - `data`: null 或错误详情
  - `traceId`: string（便于定位）

#### 2.2 错误码规范
- `1xxxx`：认证相关（未登录、Token过期、被踢、无效签名）
- `2xxxx`：权限相关（无权限、数据权限不足）
- `3xxxx`：参数校验/业务校验
- `5xxxx`：系统/依赖（DB、缓存、第三方）

#### 2.3 审计字段规范（所有业务表建议统一）
- `id`：主键（**已定：雪花ID**，`int64/bigint`，**禁止自增**；MySQL 用 `BIGINT`/`BIGINT UNSIGNED`，PostgreSQL 用 `BIGINT`）
- `created_at`、`created_by`
- `updated_at`、`updated_by`
- `deleted_at`、`deleted_by`（软删可选）
- `tenant_id`（本期不启用，但可预留）

#### 2.4 权限标识规范
- **已选方案B：权限点 = `METHOD + PATH`（推荐存储为单个字符串）**
- 规范化格式（建议）：
  - `METHOD␠/path`（METHOD 大写；path 以 `/` 开头；不带域名、不带 query）
  - 示例：`GET /admin/users`、`POST /admin/users`、`DELETE /admin/users/:id`
- 路由参数规范（必须统一，否则权限匹配会混乱）：
  - 使用 **规范路由模板** 作为权限点，而不是实际请求值
  - 例：实际请求 `GET /admin/users/123` → 权限点写成 `GET /admin/users/:id`
- 按钮权限与页面能力：
  - 按钮/菜单可绑定 **一个或多个** 权限点（字符串数组）
  - 显示规则默认 `ANY`：满足其中任意一个即可显示（需要 `ALL` 时显式声明）

#### 2.5 数据权限（行级）规则模型（建议）
- 角色数据范围 `data_scope`：
  - `ALL`：全部数据
  - `DEPT`：本部门
  - `DEPT_AND_CHILD`：本部门及子部门
  - `SELF`：本人数据（按 `created_by` 或业务字段）
  - `CUSTOM`：自定义部门集合（角色-部门关系表）
- 生效方式：
  - 对查询类接口：自动拼接过滤条件（Service/DAO 层统一处理）
  - 对导出类接口：同样生效
  - 对详情/更新/删除：必须验证行级可访问

#### 2.6 多语言（中英）规范
- 前端：基于 i18n（语言包 JSON/TS），所有菜单、按钮文案、提示信息来自语言包。
- 后端：
  - `message` 可先中文为主；第二阶段引入 `Accept-Language` 返回对应语言（后期增强）。
  - 错误码稳定，前端可用错误码映射多语言提示（建议）。

#### 2.7 主题与暗黑模式规范
- 主题配置持久化：`localStorage`（key：`themeMode`/`themeTokens`）。
- 支持：
  - `light`/`dark`
  - 主色（primary）
  - 布局密度/圆角（可选）

#### 2.8 数据库兼容策略（MySQL8 + PostgreSQL）
- **SQL 方言隔离**：尽量使用 ORM/构建器或可移植 SQL；必须用方言时，放在 `internal/dao/dialect/`，按 driver 分支。
- **主键策略（已定）**：雪花ID（`int64/bigint`），避免自增差异 + 分布式友好；各表主键、外键一律使用同一类型。
- **索引建议**：主键聚簇/BTREE；业务唯一键显式建唯一索引；时间范围查询（`created_at`）建普通索引（按模块评估）。
- **时间字段**：统一使用 UTC 存储 + 前端本地化展示（或统一时区策略写入文档）。
- **分页**：统一 `limit/offset`（两库一致）。
- **唯一约束**：在迁移脚本中显式声明（两库都支持）。

### 3. 关键设计（落地方案摘要）

#### 3.1 JWT 单点登录（SSO）与踢人
- 借鉴 HotGo 的成熟落地方式（推荐）：**JWT 作为签名载体 + 服务端缓存保存会话态**，从而天然支持“单点/踢人/主动失效/自动续期”。
- 关键点：
  - JWT 只负责签名与携带身份（`uid/app/role` 等），**真正的登录态以缓存为准**（生产建议 Redis）。
  - 登录时写入：
    - `tokenKey(app, authKey)`：token 会话（含过期时间、刷新时间、刷新次数）
    - `bindKey(app, userId)`：用户当前绑定的 tokenKey（用于单点）
  - 鉴权时校验：
    - tokenKey 必须存在且未过期
    - 若开启单点（`multiLogin=false`），则 `bindKey` 必须存在且值等于当前 tokenKey，否则判定“异地登录/被踢”
  - 踢人实现：
    - 管理员删除/覆盖 `bindKey`（最简单是删除），使旧 token 立即失效
  - 错误语义：
    - “未登录/过期”与“异地登录/被踢”必须用不同错误码，前端做不同提示与跳转

#### 3.2 动态路由（后端返回）
- 后端返回：菜单树 + 路由元信息（path、name、componentKey、icon、hideInMenu、sort、accessCodes 等）。
- 前端维护一个 `componentMap`：`componentKey -> React Component`，动态注入路由。
- 按钮权限：后端返回 `permissions: string[]`，前端用于 `Access` 控制（**每一项都是 `METHOD + PATH` 权限点**）。
  - 建议补充两类字段以提升可扩展性（借鉴 HotGo 的 `meta` 思路）：
    - `meta.permissionsAny: string[]`（满足任一即可显示）
    - `meta.permissionsAll: string[]`（必须全部满足才显示，可选）

#### 3.3 行级数据权限实现点
- 借鉴 HotGo：把“数据权限”做成 **ORM Handler 可插拔能力**，业务侧只需在查询链上加一行即可生效。
- 我们的落地方式：
  - 在 Service 层注入 `DataScopeContext`（用户/角色/部门/可见部门集合/可见用户集合）
  - 提供：
    - `FilterAuth`：自动探测常用字段（优先 `created_by`，其次 `dept_id`/`owner_id` 等，按我们规范定）
    - `FilterAuthWithField(field)`：业务自定义过滤字段
  - 对“详情/更新/删除”同样提供 `AssertCanAccess` 校验，避免只过滤列表

#### 3.4 部门树实现（已定：闭包表 Closure Table）
- 表一：`sys_dept`
  - `id`（雪花ID）
  - `parent_id`（雪花ID，根为 0 或 NULL，二选一并统一）
  - `name`、`status`、`sort` 等业务字段 + 审计字段
- 表二：`sys_dept_closure`（闭包表）
  - `ancestor_id`（雪花ID）
  - `descendant_id`（雪花ID）
  - `depth`（int，`0` 表示自身；`1` 表示父子；以此类推）
  - 约束/索引建议：
    - 主键或唯一约束：`(ancestor_id, descendant_id)`
    - 索引：`descendant_id`（用于反查祖先链），必要时再加 `ancestor_id, depth`
- 维护规则（必须事务化）：
  - **新增部门**：插入 `sys_dept` 后，向闭包表写入：自身 `(id,id,0)` + 继承父部门所有祖先链（父的所有 `ancestor_id` → 新 `descendant_id`）。
  - **移动部门（改 parent）**：需要重建该子树在闭包表中的祖先关系（删除旧祖先路径、插入新祖先路径）；这是闭包表最关键的实现点，必须写清楚并配测试用例。
  - **删除部门**：若允许删除，需先校验无子部门/或级联删除；闭包表同步清理该子树相关记录。
- 数据权限查询（DEPT_AND_CHILD）推荐写法：
  - 将“可见部门集合”定义为：`SELECT descendant_id FROM sys_dept_closure WHERE ancestor_id = :currentDeptId`
  - 再用于业务表过滤：`dept_id IN (可见部门集合)`

#### 3.5 借鉴 HotGo 的工程化设计理念（我们将落地为硬规范）
> 目的：让“系统能力”而不是“业务代码”决定一致性；为后续“生成器后置”提供稳定基座。

- **命令行多入口（强烈推荐）**：
  - 后端采用 `cmd/http` 作为主服务入口；后续扩展 `cmd/cron`、`cmd/queue`、`cmd/tools`（迁移/初始化/导入导出）。
- **路由按应用分组（admin/api/home/websocket）**：
  - 即便第一期只做 admin，也要把路由注册放到 `internal/router/admin.go` 这样的文件里；
  - 生成器未来可把生成路由挂到 `router/genrouter`，做到“生成代码不污染手写路由”。
- **中间件链顺序固定（不可随意调整）**：
  - `Ctx(i18n/trace/context初始化)` → `CORS` → `Blacklist` → `DemoLimit` → `PreFilter` → `ResponseHandler`
  - `PreFilter`：先 Parse+tag校验，再执行 `Filter()`（若实现）并回写参数（保证进入业务层前参数已规整）。
  - `ResponseHandler`：统一格式化响应；debug 输出堆栈；prod 隐藏敏感错误但保留 traceId。
- **服务接口 + 实现注入（面向接口编程）**：
  - `internal/service` 只放接口与 `RegisterXxx()`；`internal/logic` 在 `init()` 中注入实现；
  - 达到：模块解耦、实现可替换、便于单测、也便于未来插件化。
- **RBAC 推荐 Casbin（策略引擎）**：
  - 菜单表 `permissions` 字段作为权限点来源；启动/刷新时加载为 Casbin policies；
  - 角色使用 `role.key` 作为 subject；`permissions` 支持逗号分隔；动作支持 `ActionAll` 或 HTTP Method；
  - 菜单权限修改后可“刷新策略”实时生效（无需重启）。
- **生成器后置但规范前置**：
  - HotGo 把“字段类型→Go/TS类型→默认表单组件→默认校验→默认查询方式”规范化；
  - 我们也必须在写第一张业务表前固化对应规则（否则生成器无法稳定产出）。

### 5. 按天执行计划（每天 5 小时）

> 说明：每一天都按“后端/前端/产出物/验收”给任务。实际执行中允许交换顺序，但不建议跳过“规范日”。

#### 第 1 天：仓库初始化与工程基线
- 后端：
  - 初始化 GoFrame 工程骨架，确定目录结构与配置加载方式
- 前端：
  - 拉起 `art-design-pro` 工程，完成本地运行
- 产出物：
  - `server/`、`web/`、`deploy/`、`docs/README.md`
- 验收：
  - 前后端都能本地启动；README 写明启动命令与端口

#### 第 2 天：统一规范文档与接口返回框架
- 后端：
  - 统一返回结构、错误码枚举、全局异常处理
  - TraceId 注入（请求链路标识）并回传给前端
  - 参考 HotGo：落地 `PreFilter`（Req 结构体可实现 `Filter(ctx)` 做预处理/二次校验）
  - 参考 HotGo：落地 `ResponseHandler`（debug 可返回堆栈，prod 隐藏敏感错误但保留 traceId）
- 前端：
  - 请求层封装：统一处理 code/message；401/403 统一跳转与提示
- 产出物：
  - `docs/02-api-规范.md`（返回结构/错误码/分页规范）
- 验收：
  - 任意接口返回结构一致；前端能统一弹错与拦截

#### 第 3 天：数据库接入与双库兼容验证
- 后端：
  - MySQL8 + PostgreSQL 两套连接配置
  - 设计迁移策略（SQL脚本或迁移工具），跑通两库初始化
- 前端：无（预留联调）
- 产出物：
  - `server/manifest/config/config.dev.yaml`（含两库示例）
  - `server/database/migrations/`（初始化脚本）
- 验收：
  - 同一套后端代码可切换连接 MySQL/PG 并正常启动

#### 第 4 天：基础数据表 v1（系统核心）
- 后端：
  - 设计并落地系统表（用户/角色/菜单/关系/部门/岗位）
  - 审计字段统一（created_by 等）
- 前端：无
- 产出物：
  - `docs/03-表结构v1.md`
  - 初始化数据（admin、超级管理员角色、基础菜单）
- 验收：
  - 两库均可初始化并写入基础数据；能查出 admin 用户

#### 第 5 天：JWT 认证（登录/刷新/退出）后端
- 后端：
  - 登录接口、刷新接口、退出接口
  - 密码加密策略（bcrypt/argon2 选其一并写入文档）
  - 借鉴 HotGo：token 会话态写入缓存（为后续“单点/踢人”打地基）
- 前端：
  - 登录页对接准备（先 mock）
- 产出物：
  - `POST /api/auth/login`
  - `POST /api/auth/refresh`
  - `POST /api/auth/logout`
- 验收：
  - Postman：登录拿 token；刷新得到新 token；退出后 token 失效

#### 第 6 天：单点登录（SSO）与踢人能力后端
- 后端：
  - 单点策略落地（`bindKey(app,userId) -> tokenKey`），鉴权时强校验绑定关系
  - 管理员踢人接口：删除/覆盖 bindKey，使旧 token 立即失效
  - “过期/未登录”与“异地登录/被踢”错误码区分（前端提示不同）
- 前端：
  - 401/被踢提示与强制退出流程
- 产出物：
  - `POST /api/system/user/kick`
  - `docs/04-认证与会话.md`
- 验收：
  - A 端登录后，管理员踢人，A 端下一次请求立刻失效并回登录页

#### 第 7 天：前端登录链路联调
- 后端：
  - CORS/跨域策略（开发态）
- 前端：
  - token 存储（含 refresh）
  - 路由守卫：未登录跳转登录
- 产出物：
  - `frontend/src/utils/auth.ts`
  - `frontend/src/services/request.ts`
- 验收：
  - 前端可登录进入后台；刷新页面仍保持登录态

#### 第 8 天：权限模型落地（菜单+按钮）后端
- 后端：
  - 权限点模型：角色-权限点/菜单-权限点（**权限点采用 `METHOD + PATH`**）

### 7. 已确认的架构决策（已拍板）
1. **主键策略**：**雪花ID（int64/bigint）**，全库统一，禁止自增。
