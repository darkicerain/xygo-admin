### 开发规范文档（当前阶段约定汇总）

> 本文档只总结“已经在项目里落地的规范”，后续有新增再补充。  
> 目录以 `server/` 为后端、`web/` 为前端约定展开。

---

### 1. 工程结构与命名

- **后端目录**：`server/`
  - `main.go`：入口（`gctx.GetInitCtx` → `global.Init` → `cmd.Main.Run`）
  - `internal/cmd`：命令行入口（目前只有 `main`，后续可扩展 `http/cron/queue/tools`）
  - `internal/controller/{模块}`：控制器（HTTP 层）
  - `internal/model`：
    - `model/entity`：数据表实体（gf gen dao 自动生成）
    - `model/do`：DAO 绑定的 do 结构体（gf gen dao 自动生成）
    - `model/input/{模块缩写}in`：业务入参/出参模型（计划对齐 HotGo 风格）
    - `model/auth_user.go`：登录态用户信息
  - `internal/dao`：数据访问层（gf gen dao 生成）
  - `internal/middleware`：中间件（统一响应、鉴权等）
  - `internal/token`：token/JWT 相关（会话存储、解析）
  - `internal/consts`：常量与错误码（error message、code 等）
  - `internal/global`：全局初始化（时区、日志、后续扩展运行模式/Trace/缓存等）
  - `api/{模块}`：对外 API DTO（带 `g.Meta` 的 Req/Res 结构）
  - `manifest/config/config.yaml`：服务配置（端口、数据库、JWT 等）
- **前端目录**：`web/`（尚未开始改造，规划中）

命名约定：
- 后台模块统一使用 `admin` 前缀：如路由 `/admin/**`、表名 `xy_admin_*`、包名 `controller/admin`。
- 表前缀统一：`xy_`，通过 `database.default.Prefix = "xy_"` 配置。

### 2. 数据库与表设计

- **当前主库**：MySQL（`xygo`），后续兼容 PostgreSQL。
- **连接配置**：`manifest/config/config.yaml`

```yaml
database:
  default:
    link:   "mysql:xygo:******@tcp(127.0.0.1:3306)/xygo?loc=Local&parseTime=true"
    Prefix: "xy_"
    debug:  true
```

- **主键策略**：整型自增主键（`INT UNSIGNED AUTO_INCREMENT`），不再使用雪花 ID。
- **后台基础表命名**：
  - `xy_admin_dept`：部门表
  - `xy_admin_dept_closure`：部门闭包表（祖先-后代关系）
  - `xy_admin_user`：管理员表
  - `xy_admin_role`：角色表
  - `xy_admin_user_role`：管理员-角色关联
  - `xy_admin_menu`：菜单/权限点
  - `xy_admin_role_menu`：角色-菜单关联

MySQL 为主，PostgreSQL 兼容时注意点：
- 自增列映射为 `SERIAL/BIGSERIAL`。
- 时间字段统一使用 `TIMESTAMP/DATETIME`，并在 Go 层用 `*gtime.Time`。

---

### 3. API DTO 与业务模型分层（对齐 HotGo 风格）

目标：**API 层只关心 HTTP 协议，业务入参/出参放在 internal 模型层**。

约定：
- 所有对外接口的 Req/Res 结构体放在 `server/api/{模块}/` 下，只负责：
  - `g.Meta`（path/method/tags/summary）
  - 内嵌 `internal/model/input/{模块缩写}in` 里的业务入参/出参模型。
- 业务入参/出参结构体统一放在：
  - `internal/model/input/adminin`：后台 admin 模块的输入/输出模型
  - 命名例子：
    - `AdminUserListInp` / `AdminUserListModel`
    - `AdminUserEditInp` / `AdminUserViewModel`

控制器层（`internal/controller/{模块}`）优先依赖 `adminin.*`，不直接依赖 api 包里的 Req/Res。  
目前项目中已有的简单接口（登录/profile 等）暂时直接使用 api 包结构，后续在模块演进时逐步迁移到上述模式。

### 4. 认证与登录态

#### 4.1 JWT + 会话存储（当前实现）

- 配置（`manifest/config/config.yaml`）：

```yaml
auth:
  jwt:
    secret:  "xygo123456"   # 签名密钥（生产环境必须替换）
    expires: 7200           # accessToken 有效期（秒）
```

- 生成与解析逻辑：
  - `internal/token/jwt.go`：
    - `Generate(ctx, model.AuthUser) (accessToken, expiresIn, err)`：
      - 生成 JWT（HS256），载荷包含 `userId/username`、签发时间、过期时间。
      - 写入会话存储（`internal/token/store.go`），实现单点登录（同一用户旧 token 会被删除）。
    - `Parse(ctx, accessToken) (*model.AuthUser, error)`：
      - 校验签名与过期时间。
      - 确认会话在存储中存在（支持踢下线/注销后失效）。
- 会话存储：
  - 当前实现：文件存储 `storage/token/sessions.json`（通过 `token.Save/Get/Delete/DeleteByUser` 操作）。
  - 未来规划：替换为 Redis 驱动（实现相同接口），保证代码调用保持一致。

#### 4.2 登录相关接口

- `POST /admin/auth/login`：
  - 请求体：`username/password`（form 或 x-www-form-urlencoded）。
  - 控制器：`internal/controller/admin.Login`。
  - 行为：
    - 按用户名查 `xy_admin_user`。
    - 密码校验：`md5(password + salt)`。
    - 调用 `token.Generate` 生成 `accessToken`，返回：
      - `id/username/nickname/accessToken/expiresIn`。
- `POST /admin/auth/logout`：
  - 控制器：`internal/controller/admin.Logout`。
  - 从 `Authorization: Bearer xxx` 头中取 token，调用 `token.Delete` 删除会话。
- `GET /admin/auth/profile`：
  - 控制器：`internal/controller/admin.Profile`。
  - 从 `Authorization` 解析 token，并返回当前登录管理员信息，**响应结构与前端约定强绑定**：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "id": 1,
    "username": "admin",
    "nickname": "超级管理员",
    "isSuper": true,
    "roles": ["R_SUPER"],
    "buttons": []
  },
  "traceId": "xxxx"
}
```

- 说明：
  - `roles: string[]`：**角色编码列表**，用于前端菜单/路由权限控制。
  - `buttons: string[]`：**按钮权限标识列表**，预留给后续 RBAC 按钮级权限（当前返回空数组）。
  - 字段定义落地在：`internal/model/input/adminin/auth.go::ProfileModel`，后续涉及登录用户信息的扩展一律从此处补充。

### 5. 鉴权中间件与路由设计

- 所有路由注册集中在 `internal/cmd/cmd.go`：

```go
s.Group("/", func(group *ghttp.RouterGroup) {
    // 全局统一响应
    group.Middleware(middleware.ResponseHandler)

    // 无鉴权接口：hello/system
    group.Bind(hello.NewV1())
    group.Bind(system.NewV1())

    // admin 后台接口（带鉴权）
    group.Group("/", func(ag *ghttp.RouterGroup) {
        ag.Middleware(middleware.AdminAuth)
        ag.Bind(admin.NewV1())
    })
})
```

- `middleware.AdminAuth` 逻辑（`internal/middleware/auth.go`）：
  - 放行路径：`/admin/auth/login`（无需 token）。
  - 其它 `/admin/**`：
    - 从 `Authorization` 头解析 `Bearer {token}`。
    - 使用 `token.Parse` 校验签名、过期与会话存在性。
    - 失败时设置错误码 `CodeNotAuthorized` 并中止后续处理。

### 6. 统一响应结构与错误处理

- 中间件：`internal/middleware/response.go`。
- 统一响应结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {...},
  "traceId": "xxxx"
}
```

- 成功：
  - `code = 0`（`consts.CodeOK`）
  - `message = "ok"`
  - `data = Handler 返回的 Res 结构体`
- 失败：
  - 从 `r.GetError()` 中获取错误，取 `gerror.Code(err)` 作为 `code`，`gerror.Current(err).Error()` 作为 `message`。
  - 如果未设置业务码，默认 `code = -1`（`consts.CodeFailed`）。
  - `traceId` 始终返回，便于日志追踪。
- 错误消息二次包装：
  - 在 `internal/consts/error.go` 中提供 `ErrorMessage(err)`，用于在非 debug 模式隐藏内部错误细节，返回统一友好文案。

#### 6.1 HTTP Status 与业务 code 对齐规则

- **整体原则**：前端主要依赖响应 JSON 中的 `code` 字段做业务判断，HTTP 状态码仅用于网络级/路由级异常。
- **HTTP 状态码约定：**
  - `200`：接口路由存在，后端成功处理请求（无论业务成功/失败）；业务结果由 `code` 决定。
  - `404`：路由未匹配（路径或方法写错），一般视为前后端联调 bug，不作为正常业务分支；返回体中 `code = consts.CodeNotFound.Code()`。
  - `5xx`：仅保留给 GoFrame 或中间件未捕获的 panic/系统级异常（理想情况下应趋近于 0）。
- **JSON `code` 字段与前端行为：**
  - `0`：成功，前端正常展示数据。
  - `61`（`consts.CodeNotAuthorized.Code()` 当前实际值）：未登录/会话失效/被踢下线。
    - 前端行为：触发统一未登录处理（弹出“登录失效，请重新登录”提示，清理本地状态并跳转登录页）。
  - `consts.CodeNoPermission.Code()`：已登录但无接口/资源权限（403 语义）。
    - 前端行为：可跳到 403 页面或弹出“无访问权限”提示。
  - `>= 10001`：业务错误（参数错误、数据不存在、重复数据等），由具体业务决定提示文案。
- **实现要点：**
  - 所有已注册的 API 路由，控制器层**不主动写入非 200 的 HTTP 状态码**，统一交给 `ResponseHandler` 输出 JSON。
  - 路由不存在时，由 GoFrame 返回 404，`ResponseHandler` 包装为统一 JSON 结构并设置 `code = consts.CodeNotFound.Code()`。
  - 前端 `web/src/utils/http` 中的 `ApiStatus` 与上述 `code` 保持一致，目前已约定：
    - `success = 0`
    - `unauthorized = 61`
  - 新增业务错误码时，需同时在 `internal/consts/code.go` 注释清楚语义，避免魔法数字散落在业务代码中。

### 7. 错误码规范（code 字段）

- 实现入口：`internal/consts/code.go`。
- 约定：
  - `0`：成功（`consts.CodeOK`）。
  - `< 0`：GoFrame 内置错误码（不推荐业务层直接使用数字）。
  - 常见通用错误复用 gcode：
    - `consts.CodeFailed = gcode.CodeNil`：通用失败。
    - `consts.CodeNotAuthorized = gcode.CodeNotAuthorized`：未登录/登录失效。
    - `consts.CodeNoPermission = gcode.CodeSecurityReason`：无访问权限。
    - `consts.CodeNotFound = gcode.CodeNotFound`：路由/资源不存在。
  - 业务自定义错误码从 `10001` 起：
    - `consts.CodeInvalidParam (10001)`：参数错误。
    - `consts.CodeDuplicateData (10002)`：数据已存在。
    - `consts.CodeDataNotFound (10003)`：业务数据不存在。
- 使用方式：

```go
import "xygo/internal/consts"

return nil, gerror.NewCode(consts.CodeInvalidParam, "用户名不能为空")
```

#### 7.1 错误码使用规范（禁止硬编码）

- **唯一来源**：所有对外可见的业务错误码必须在 `internal/consts/code.go` 中集中定义，业务代码只能引用 `consts.CodeXXX`，**禁止直接写具体数字或直接引用 `gcode.CodeXXX`**。
- **服务端示例：**
  - ✅ 推荐：
    ```go
    import "xygo/internal/consts"

    // 未登录
    return nil, gerror.NewCode(consts.CodeNotAuthorized, "未登录")

    // 数据不存在
    return nil, gerror.NewCode(consts.CodeDataNotFound, "角色不存在")
    ```
  - ❌ 禁止：
    ```go
    // 直接写数字或直接用 gcode
    return nil, gerror.NewCode(gcode.CodeNotAuthorized, "未登录")
    return nil, gerror.NewCode(gcode.New(10003, "", nil), "角色不存在")
    ```
- **前端对应关系**：
  - 由于前端无法直接引用 Go 代码，前端只在 `web/src/utils/http/status.ts` 中定义与后端一致的常量映射（例如：`success = 0`、`unauthorized = 61`），其他业务错误码一律按文档说明通过 `code` 值做展示，不在前端硬编码判断。

### 8. 权限与路由命名（预留规范）

当前已选方案：**权限点 = `METHOD + PATH`**，后续所有 RBAC 相关字段都按此规范存储：
- 格式：`"GET /admin/user/list"`、`"POST /admin/auth/login"` 等。
- 菜单表中建议使用 JSON 数组字段保存权限点列表。
- 统一存放在 `xy_admin_menu.perms` 字段中（后续在权限模块落地时细化）。

#### 8.1 角色编码规范（与前端对齐）

- 角色编码来源：`xy_admin_role.key`，**必须与前端路由/菜单里的 `roles: []` 保持一致**。
- 约定的基础角色（可扩展）：
  - `R_SUPER`：超级管理员（通常拥有所有菜单和权限）。
  - `R_ADMIN`：后台管理员（业务可控权限集合）。
  - `R_USER`：普通用户（如未来接 user 端，可复用该约定）。
- 登录成功后，当前用户拥有的角色编码通过 `/admin/auth/profile` 返回：
  - 查询链路：`xy_admin_user` → `xy_admin_user_role` → `xy_admin_role.key`。
  - 响应字段：`data.roles: string[]`。
- 前端使用方式（art-design-pro 内置用法）：
  - 路由 meta 中配置 `roles: ['R_SUPER', 'R_ADMIN']`，表示**任一角色具备即可访问**。
  - 登录后前端根据 `roles` 数组和路由 meta 自动过滤菜单/路由，无需再做额外映射。

#### 8.2 按钮权限响应规范（预留）

- 统一通过 `Profile` 或专门的权限接口返回：
  - `buttons: string[]`：按钮权限标识列表，推荐同样使用 `METHOD + PATH` 或业务别名（如 `system:user:add`）。
- 当前阶段：
  - 字段已在 `ProfileModel` 和前端 `Api.Auth.UserInfo` 中预留。
  - 具体填充值将在“菜单 + 权限点 + 角色绑定”阶段一并落地，并同步补充本规范。
