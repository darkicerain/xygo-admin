# 权限改造方案（基于 HotGo 实现）

> 本文档基于 HotGo 项目的实际权限实现，为当前 XYGo 项目定制改造方案。

## 一、当前状态分析（XYGo vs HotGo）

### 1.1 数据库表结构对比

#### admin_role（角色表）

| 字段 | XYGo | HotGo | 说明 |
|------|------|-------|------|
| id | ✅ | ✅ | 角色ID |
| name | ✅ | ✅ | 角色名称 |
| key | ✅ | ✅ | 角色标识 |
| data_scope | ✅ | ✅ | 数据范围（已有字段） |
| custom_depts | ✅ | ✅（custom_dept） | 自定义部门权限 |
| pid | ✅ | ✅ | 上级角色 |
| level | ✅ | ✅ | 关系树等级 |
| tree | ✅ | ✅ | 关系树路径 |

**结论**：XYGo 的 `admin_role` 表结构已完整，字段齐全。

#### admin_user（用户表）

| 字段 | XYGo | HotGo（admin_member） | 说明 |
|------|------|---------------------|------|
| id | ✅ | ✅ | 用户ID |
| dept_id | ✅ | ✅ | 部门ID |
| created_by | ✅ | ✅ | 创建人（用于数据权限过滤） |
| pid | ❌ | ✅ | **缺失**：上级用户ID（用于"自己和下级"数据范围） |

**结论**：XYGo 用户表缺少 `pid`（上级ID）字段，需要添加以支持"自己和直属下级"、"自己和全部下级"数据范围。

#### admin_dept（部门表）

XYGo 已有完整的部门表，包含树形结构支持。

### 1.2 代码实现对比

| 功能模块 | XYGo | HotGo | 差异 |
|---------|------|-------|------|
| 数据范围常量定义 | ❌ | ✅ | HotGo 在 `internal/consts/role.go` 定义了 7 种数据范围 |
| 数据权限 Handler | ❌ | ✅ | HotGo 有 `FilterAuth` Handler 自动过滤数据 |
| 多租户 Handler | ❌ | ✅ | HotGo 有 `FilterTenant` Handler 支持租户/商户/用户隔离 |
| 用户上下文（Identity） | ⚠️ | ✅ | XYGo 有但可能不完整（需确认是否含 DeptId、Pid、RoleKey） |
| 前端数据权限 UI | ❌ | ✅ | HotGo 有编辑数据权限的对话框组件 |
| 按钮权限指令 | ✅ | ✅ | XYGo 已有 v-auth/v-roles |
| 字段级权限 | ❌ | ❌ | 双方均未实现 |

## 二、改造目标

基于 HotGo 的成熟实现，为 XYGo 添加完整的数据权限控制：

1. ✅ **菜单/页面级权限**（已有，继续保留）
2. ✅ **按钮/操作级权限**（已有，继续保留）
3. ✨ **数据范围权限**（新增，7 种数据范围）
4. ✨ **多租户支持**（可选，基于部门类型）
5. ✨ **字段级权限**（扩展，HotGo 也未实现）

#### 1.2 确认其他表有 `created_by` 字段



```go
// FilterAuth 过滤数据权限
// 通过上下文中的用户角色权限和表中是否含有需要过滤的字段附加查询条件
func FilterAuth(m *gdb.Model) *gdb.Model {
	var (
		needAuth    bool
		filterField string
		fields      = convert.EscapeFieldsToSlice(m.GetFieldsStr())
	)

	// 优先级：created_by > member_id
	if gstr.InArray(fields, "created_by") {
		needAuth = true
		filterField = "created_by"
	}

	if !needAuth && gstr.InArray(fields, "member_id") {
		needAuth = true
		filterField = "member_id"
	}

	if !needAuth {
		return m
	}
	return m.Handler(FilterAuthWithField(filterField))
}

// FilterAuthWithField 过滤数据权限，设置指定字段
func FilterAuthWithField(filterField string) func(m *gdb.Model) *gdb.Model {
	return func(m *gdb.Model) *gdb.Model {
		var (
			role *entity.AdminRole

		if user == nil {
			return m
		}

		// 查询用户角色
		err := dao.AdminRole.Ctx(ctx).Where(dao.AdminRole.Columns().Id, user.RoleId).Scan(&role)
		if err != nil {
			g.Log().Panicf(ctx, "failed to role information err:%+v", err)
		}

		if role == nil {
			g.Log().Panic(ctx, "failed to role information roleModel == nil")
		}

		// 超管拥有全部权限
		if role.Key == consts.SuperRoleKey {
			return m
		}

			ds, err := dao.AdminUser.Ctx(ctx).Fields(dao.AdminUser.Columns().Id).Where(dao.AdminUser.Columns().DeptId, in).Array()

			}

		}

			// 假设 role.CustomDepts 是 JSON 字符串存储的 []int
			var customDepts []int64
			if role.CustomDepts != "" {
				// 解析 JSON，根据实际存储格式调整
				// ... 解析 role.CustomDepts 到 customDepts
			}
			m = m.WhereIn(filterField, getDeptIds(customDepts))
		case consts.RoleDataSelf: // 仅自己
			m = m.Where(filterField, user.Id)
		case consts.RoleDataSelfAndSub: // 自己和直属下级
			m = m.WhereIn(filterField, GetSelfAndSub(ctx, user.Id))
		case consts.RoleDataSelfAndAllSub: // 自己和全部下级

		default:
			g.Log().Panic(ctx, "dataScope is not registered")
		}
		return m
	}
```

#### 4.2 创建 `FilterTenant` Handler（可选，多租户）

**文件**：`server/internal/library/hgorm/handler/filter_tenant.go`

参考 HotGo 实现：

```go
package handler

import (
	"github.com/gogf/gf/v2/database/gdb"
	"github.com/gogf/gf/v2/text/gstr"
	"xygo/utility/convert"
)

// FilterTenant 过滤多租户数据权限
// 根据部门类型识别当前租户、商户、用户身份，过滤只属于自己的数据
func FilterTenant(m *gdb.Model) *gdb.Model {
	var (
		needAuth    bool
		filterField string
		fields      = convert.EscapeFieldsToSlice(m.GetFieldsStr())
		ctx         = m.GetCtx()
		user        = GetUserFromContext(ctx)
	)

	if user == nil {
		return m
	}

	// 租户
	if user.DeptType == "tenant" && gstr.InArray(fields, "tenant_id") {
		needAuth = true
		filterField = "tenant_id"
	}

	// 商户
	if user.DeptType == "merchant" && gstr.InArray(fields, "merchant_id") {
		needAuth = true
		filterField = "merchant_id"
	}

	// 用户
	if user.DeptType == "user" && gstr.InArray(fields, "user_id") {
		needAuth = true
		filterField = "user_id"
	}

	if !needAuth {
		return m
	}

	m = m.Where(filterField, user.Id)
	return m
}
```

```go
func (c *ControllerV1) UserList(ctx context.Context, req *api.UserListReq) (res *api.UserListRes, err error) {
```

### 步骤 9：字段级权限实现（核心功能）

字段级权限用于控制表单/表格中的字段显示和编辑权限，是精细化权限控制的重要组成部分。

#### 9.1 数据库设计

##### 9.1.1 角色表添加字段权限配置

```sql

ALTER TABLE `admin_role` 
ADD COLUMN `field_perms` JSON COMMENT '字段权限配置（JSON，格式见下方说明）' AFTER `custom_depts`;
```

##### 9.1.2 创建字段权限配置表（推荐方案）

为便于管理和扩展，建议创建独立的字段权限配置表：

```sql
-- 字段权限配置表
CREATE TABLE `admin_field_perm` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  `role_id` BIGINT UNSIGNED NOT NULL COMMENT '角色ID',
  `module` VARCHAR(50) NOT NULL COMMENT '模块名称（如：system、org）',
  `resource` VARCHAR(100) NOT NULL COMMENT '资源标识（表名或页面标识，如：admin_user、user_list）',
  `field_name` VARCHAR(100) NOT NULL COMMENT '字段名称（如：mobile、salary）',
  `field_label` VARCHAR(100) DEFAULT NULL COMMENT '字段显示名称',
  `perm_type` TINYINT NOT NULL DEFAULT 0 COMMENT '权限类型：0=不可见，1=只读，2=可编辑',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态：0=禁用，1=启用',
  `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注',
  `create_time` INT UNSIGNED NOT NULL COMMENT '创建时间',
  `update_time` INT UNSIGNED NOT NULL COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_role_resource_field` (`role_id`, `resource`, `field_name`),
  KEY `idx_role_id` (`role_id`),
  KEY `idx_resource` (`resource`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='字段权限配置表';
```

**权限类型说明**：
- `0`：不可见（字段隐藏，前端不显示）
- `1`：只读（字段可见但不可编辑，表单项禁用）
- `2`：可编辑（字段可见且可编辑，完全权限）

##### 9.1.3 字段权限规则优先级

```text
1. 超级管理员：所有字段可见可编辑（跳过字段权限检查）
2. 明确配置：按 admin_field_perm 表配置的权限
3. 默认规则：未配置的字段默认可见可编辑（或根据业务需求设为不可见）
```

```go
	_, err = dao.AdminFieldPerm.Ctx(ctx).
```

```typescript
import type { Directive, DirectiveBinding } from 'vue'
import { useUserStore } from '@/store/modules/user'

interface FieldBinding {
  resource: string  // 资源标识（表名或页面）
  field: string     // 字段名
}

// v-field-visible：控制字段可见性
export const fieldVisible: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding<FieldBinding>) {
    const userStore = useUserStore()
    const { resource, field } = binding.value
    
    if (!userStore.isFieldVisible(resource, field)) {
      // 移除元素
      el.parentNode?.removeChild(el)
    }
  }
}

// v-field-editable：控制字段可编辑性
export const fieldEditable: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding<FieldBinding>) {
    const userStore = useUserStore()
    const { resource, field } = binding.value
    
    if (!userStore.isFieldEditable(resource, field)) {
      // 设置为只读/禁用
      setElementReadonly(el, true)
    }
  },
  updated(el: HTMLElement, binding: DirectiveBinding<FieldBinding>) {
    const userStore = useUserStore()
    const { resource, field } = binding.value
    
    const editable = userStore.isFieldEditable(resource, field)
    setElementReadonly(el, !editable)
  }
}

// 辅助函数：设置元素为只读
function setElementReadonly(el: HTMLElement, readonly: boolean) {
  // 查找 input/textarea/select 元素
  const inputs = el.querySelectorAll('input, textarea, select')
  
  inputs.forEach((input) => {
    if (readonly) {
      input.setAttribute('disabled', 'true')
      input.setAttribute('readonly', 'true')
      // 添加只读样式
      input.classList.add('is-readonly-by-perm')
    } else {
      input.removeAttribute('disabled')
      input.removeAttribute('readonly')
      input.classList.remove('is-readonly-by-perm')
    }
  })
  
  // 如果是 ElButton 等操作按钮，也禁用
  const buttons = el.querySelectorAll('button')
  buttons.forEach((btn) => {
    if (readonly) {
      btn.setAttribute('disabled', 'true')
      btn.classList.add('is-readonly-by-perm')
    } else {
      btn.removeAttribute('disabled')
      btn.classList.remove('is-readonly-by-perm')
    }
  })
}

// v-field：综合指令（同时控制可见和可编辑）
export const field: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding<FieldBinding>) {
    const userStore = useUserStore()
    const { resource, field } = binding.value
    
    // 先检查可见性
    if (!userStore.isFieldVisible(resource, field)) {
      el.parentNode?.removeChild(el)
      return
    }
    
    // 再检查可编辑性
    if (!userStore.isFieldEditable(resource, field)) {
      setElementReadonly(el, true)
    }
  }
}
```

## 五、注意事项与最佳实践

### 5.1 超级管理员判断逻辑（重要！）

#### 5.1.1 判断依据

超级管理员的判断**不是基于角色 ID**，而是基于以下字段：

| 判断方式 | 字段 | 说明 | 优先级 |
|---------|------|------|--------|
| ✅ **用户级别** | `admin_user.is_super` | 直接标记用户是否为超管 | 最高 |
| ✅ **角色级别** | `admin_role.key` | 角色的唯一标识符（如 'super'） | 次高 |
| ❌ **错误方式** | `admin_role.id` | 角色 ID 会变化，不可靠 | 禁止 |

#### 5.1.2 为什么不能用角色 ID？

```text
❌ 错误示例：
if (role.id == 1) {  // 角色 ID 可能变化
    // 超级管理员
}

✅ 正确示例：
if (user.IsSuper == 1 || role.Key == "super") {
    // 超级管理员
}
```

**原因**：
- 角色 ID 会因数据迁移、删除重建等原因变化
- 不同环境（开发/测试/生产）的角色 ID 可能不同
- ID 是自增主键，没有语义，不适合做业务判断

### 5.2 Handler 使用原则

- ✅ **查询操作**：统一使用 `Handler(handler.FilterAuth)` 过滤
- ✅ **写入/更新/删除操作**：需先查询验证权限，再执行操作
- ❌ **统计/聚合查询**：谨慎使用 Handler，避免统计数据不准确

- 敏感操作（删除、导出）需二次鉴权
- 数据权限配置变更应记录审计日志
- 定期检查数据权限配置的合理性

## 七、FAQ

### Q1：为什么不用 RBAC 的权限码控制数据权限？

**A**：数据权限与操作权限是两个维度：
- **操作权限**（菜单/按钮）：用户能做什么
- **数据权限**（数据范围）：用户能看到哪些数据

HotGo 的数据范围方案更灵活，适合部门/代理商/多租户等场景。

### Q2：数据权限会影响性能吗？

**A**：会有一定影响，但可优化：
- 为过滤字段建立索引
- 缓存部门树、用户树
- 超管跳过 Handler
- 避免在循环中使用 Handler

### Q3：如何处理跨部门协作场景？

**A**：可使用"自定义部门"数据范围，为角色指定多个可见部门。

### Q4：字段级权限如何处理关联查询？

**A**：建议：
- 后端裁剪：在 DTO 层过滤敏感字段
- 前端控制：基于字段权限配置显示/隐藏

## 十、系统权限模型详解（核心设计）

### 10.1 基础概念：用户-角色-部门-岗位关系

#### 实体关系图

```
用户（User）
  ├─ 1:1 → 部门（Dept）        [dept_id]
  ├─ N:M → 角色（Role）        [admin_user_role 中间表]
  ├─ N:M → 岗位（Post）        [admin_user_post 中间表]
  └─ 1:N → 直属下级用户         [pid 字段]

角色（Role）
  ├─ 1:1 → 数据范围配置         [data_scope, custom_depts]
  ├─ N:M → 菜单权限            [admin_role_menu 中间表]
  ├─ N:M → 字段权限配置         [admin_field_perm 表]
  └─ 树形结构（支持角色继承）     [pid, level, tree]

部门（Dept）
  └─ 树形结构                 [parent_id + 闭包表]

岗位（Post）
  └─ 独立实体（用于标识用户在部门中的职位）
```

#### 四个实体的职责

| 实体 | 职责 | 示例 | 关系类型 |
|------|------|------|---------|
| **用户** | 系统的操作主体 | 张三、李四 | 人 |
| **角色** | 权限的载体，决定能做什么 | 系统管理员、财务主管、开发工程师 | 权限集合 |
| **部门** | 组织架构，决定数据隔离范围 | 技术部、财务部、人事部 | 组织树 |
| **岗位** | 职位标识，业务属性 | CEO、产品经理、测试工程师 | 标签 |

### 10.2 权限的三个维度

#### 1. 菜单权限（功能权限）

**问题**：用户能看到哪些菜单？能访问哪些页面？

**控制方式**：
```
用户 → 角色 → 菜单
```

**权限判断流程**：
1. 用户登录，获取其所有角色
2. 查询这些角色绑定的菜单ID（`admin_role_menu` 表）
3. 根据菜单ID查询菜单列表
4. 前端动态生成路由和侧边栏

**示例**：
```
用户A：张三
  → 角色：[系统管理员, 开发工程师]
    → 菜单：[用户管理, 角色管理, 菜单管理, 系统配置, ...]
  
用户B：李四
  → 角色：[财务主管]
    → 菜单：[财务报表, 发票管理, ...]
```

#### 2. 数据权限（数据范围）

**问题**：用户能看到哪些数据记录？

**7种数据范围**：

| DataScope | 名称 | 能看到的数据 | SQL示例 |
|-----------|------|------------|---------|
| 1 | 全部权限 | 所有数据 | 无WHERE条件 |
| 2 | 当前部门 | 本部门数据 | `WHERE dept_id = 用户部门ID` |
| 3 | 部门及子部门 | 本部门+子部门 | `WHERE dept_id IN (部门ID列表)` |
| 4 | 自定义部门 | 指定部门 | `WHERE dept_id IN (自定义列表)` |
| 5 | 仅自己 | 自己创建的 | `WHERE created_by = 用户ID` |
| 6 | 自己和直属下级 | 自己+直接下级 | `WHERE created_by IN (自己, 下级列表)` |
| 7 | 自己和全部下级 | 自己+所有递归下级 | `WHERE created_by IN (自己, 所有下级)` |

**自动过滤**：通过 `FilterAuth` Handler 在SQL层自动注入WHERE条件

#### 3. 字段权限（字段级控制）

**三种权限类型**：
1. **不可见**（permType=1）：字段隐藏
2. **只读**（permType=2）：显示但不可编辑  
3. **可编辑**（permType=3）：完全可操作

### 10.3 实际业务场景示例

#### 场景1：部门主管查看用户列表

**用户**：张三，技术部主管，DataScope=2

**访问**：`/user/list`

**SQL**：
```sql
SELECT * FROM admin_user WHERE dept_id = 10  -- 技术部
```

**结果**：只看到技术部用户

#### 场景2：总监查看跨部门数据

**用户**：李四，研发总监，DataScope=3

**访问**：`/user/list`

**SQL**：
```sql
SELECT * FROM admin_user WHERE dept_id IN (5,10,11,12)  -- 研发中心+子部门
```

**结果**：看到研发中心及下属所有部门用户

### 10.4 权限矩阵总览

| 维度 | 控制对象 | 粒度 | 配置位置 | 判断位置 |
|------|---------|------|---------|---------|
| **菜单权限** | 页面/功能 | 菜单级 | 角色→菜单权限 | 路由守卫 |
| **数据权限** | 数据记录 | 行级 | 角色→数据权限 | FilterAuth |
| **字段权限** | 表单字段 | 列级 | 角色→字段权限 | 前端渲染 |

### 10.5 特殊角色：超级管理员

**判断**：`role.key = 'super_admin'`

**特权**：
- ✅ 所有菜单（需手动分配）
- ✅ 所有数据（自动豁免FilterAuth）
- ✅ 所有字段（自动豁免字段权限）

### 10.6 权限设计核心原则

```
┌─────────────────────────────────────────┐
│          用户（User）                     │
│  - id, username, dept_id, pid           │
└──────────┬──────────────────────────────┘
           │ N:M
           ↓
┌─────────────────────────────────────────┐
│          角色（Role）                     │
│  - id, name, key, pid (树形)            │
│  - data_scope (数据范围 1-7)            │
│  - custom_depts (自定义部门列表)         │
└──────────┬──────────────────────────────┘
           │
           ├─ N:M → 菜单（Menu）         【菜单权限】
           └─ N:M → 字段配置             【字段权限】

部门（Dept）：树形结构，数据隔离边界
岗位（Post）：业务属性，不参与权限判断
```

**核心思想**：
1. **权限由角色承载**：用户通过角色获得权限
2. **部门是数据边界**：数据范围基于部门树隔离
3. **三维度独立**：菜单、数据、字段权限互不影响
4. **超管豁免所有**：`super_admin` 标识判断

## 十一、数据范围配置详解（DataScope 作用说明）

### 11.1 什么是数据范围？

**数据范围（DataScope）** 是角色的核心权限配置之一，决定了**该角色的用户能看到哪些数据记录**。

### 11.2 为什么需要数据范围？

**问题场景**：
- 公司有100个员工，分属10个部门
- 技术部主管登录系统，查看用户列表
- **问题**：他应该看到所有100个员工，还是只看到技术部的员工？

**解决方案**：通过数据范围控制！
- 如果角色配置为"全部数据"（DataScope=1）→ 看到所有100个员工
- 如果角色配置为"当前部门"（DataScope=2）→ 只看到技术部的员工

### 11.3 七种数据范围详解

#### 1. 全部数据（DataScope = 1）

**作用**：能看到系统中的所有数据，无任何限制。

**适用角色**：
- 超级管理员（自动）
- 总经理、董事长
- 数据分析师（需要查看全局数据）

**SQL效果**：
```sql
SELECT * FROM admin_user;  -- 无WHERE条件限制
```

**示例**：
```
总经理登录 → 查看用户列表 → 看到所有部门的所有员工
```

---

#### 2. 当前部门（DataScope = 2）

**作用**：只能看到**自己所属部门**创建的数据。

**适用角色**：
- 部门主管
- 部门经理

**SQL效果**：
```sql
-- 假设用户属于"技术部"（dept_id=10）
SELECT * FROM admin_user WHERE dept_id = 10;
```

**示例**：
```
技术部主管（dept_id=10）登录
  → 查看用户列表
  → 只看到技术部的员工（dept_id=10）
  → 看不到财务部、人事部的员工
```

---

#### 3. 当前部门及子部门（DataScope = 3）

**作用**：能看到**自己部门 + 所有下级子部门**的数据。

**适用角色**：
- 事业部总监
- 大部门经理

**SQL效果**：
```sql
-- 假设用户属于"研发中心"（dept_id=5），下辖技术部(10)、测试部(11)
SELECT * FROM admin_user WHERE dept_id IN (5, 10, 11);
```

**示例**：
```
研发中心总监（dept_id=5）登录
  → 查看用户列表
  → 看到：研发中心 + 技术部 + 测试部 的所有员工
  → 看不到：财务部、人事部的员工
```

---

#### 4. 自定义部门（DataScope = 4）

**作用**：能看到**指定若干个部门**的数据（跨部门协作）。

**适用角色**：
- 项目经理（负责跨部门项目）
- 特殊协作角色

**配置方式**：
```json
{
  "dataScope": 4,
  "customDepts": "[10, 15, 20]"  // 技术部、市场部、客服部
}
```

**SQL效果**：
```sql
SELECT * FROM admin_user WHERE dept_id IN (10, 15, 20);
```

**示例**：
```
项目经理（配置了技术部、市场部、客服部）登录
  → 查看用户列表
  → 看到这三个部门的员工
  → 看不到其他部门的员工
```

---

#### 5. 仅自己（DataScope = 5）

**作用**：只能看到**自己创建**的数据。

**适用角色**：
- 普通员工
- 实习生

**SQL效果**：
```sql
-- 假设用户ID=100
SELECT * FROM project WHERE created_by = 100;
```

**示例**：
```
普通员工（user_id=100）登录
  → 查看项目列表
  → 只看到自己创建的项目（created_by=100）
  → 看不到同事创建的项目
```

---

#### 6. 自己和直属下级（DataScope = 6）

**作用**：能看到**自己 + 直接汇报给自己的人**创建的数据。

**适用角色**：
- 小组长
- 团队Leader

**前提**：用户表有 `pid` 字段（上级ID）

**SQL效果**：
```sql
-- 假设用户ID=100，直属下级有[101, 102, 103]
SELECT * FROM project WHERE created_by IN (100, 101, 102, 103);
```

**示例**：
```
小组长（user_id=100，下级：[101, 102, 103]）登录
  → 查看任务列表
  → 看到：自己 + 3个直属下级的任务
  → 看不到：其他组员或上级的任务
```

---

#### 7. 自己和全部下级（DataScope = 7）

**作用**：能看到**自己 + 所有递归下级**创建的数据。

**适用角色**：
- 大部门经理
- 分管副总

**前提**：用户表有 `pid` 字段

**SQL效果**：
```sql
-- 递归查询所有下级（可能有多层）
-- 假设下级树：100 → 101 → 104
--                  → 102 → 105
--                  → 103
SELECT * FROM project WHERE created_by IN (100, 101, 102, 103, 104, 105);
```

**示例**：
```
部门经理（user_id=100）登录
  ├─ 直属下级：[101, 102, 103]
  └─ 间接下级：[104, 105]（101和102的下级）
  
  → 查看项目列表
  → 看到：自己 + 所有直接和间接下级的项目
  → 实现完整的管理范围覆盖
```

### 11.4 数据范围的实际应用

#### 场景1：分级管理

```
总经理（DataScope=1）→ 看到所有数据
  ├─ 研发总监（DataScope=3）→ 看到研发中心及下属部门
  │   ├─ 技术部主管（DataScope=2）→ 只看技术部
  │   └─ 测试部主管（DataScope=2）→ 只看测试部
  └─ 财务总监（DataScope=3）→ 看到财务部门及下属
```

每个层级的管理者看到的数据范围逐级收窄。

#### 场景2：跨部门协作

```
项目经理（DataScope=4，自定义部门=[技术部, 产品部, 设计部]）
  → 负责跨部门项目
  → 能看到这三个部门的数据
  → 方便协调资源
```

#### 场景3：员工自助服务

```
普通员工（DataScope=5）
  → 查看报销单
  → 只看到自己提交的报销单
  → 保护隐私，数据隔离
```

### 11.5 数据范围的自动生效机制

**关键代码**：`FilterAuth` Handler

```go
// 用户访问列表接口时
model := dao.AdminUser.Ctx(ctx).Handler(handler.FilterAuth)

// FilterAuth 会自动根据用户的角色DataScope注入WHERE条件
// 开发者无需手动写权限过滤逻辑！
```

**流程**：
1. 用户访问 `/user/list`
2. 中间件注入用户上下文（包含角色信息）
3. Controller 调用 `Handler(FilterAuth)`
4. FilterAuth 读取角色的 `data_scope` 字段
5. 根据 DataScope 值，自动拼接 WHERE 条件
6. 返回过滤后的数据

### 11.6 配置建议

#### 推荐配置模板

| 角色类型 | 建议 DataScope | 说明 |
|---------|---------------|------|
| 超级管理员 | 1（全部数据） | 系统管理员 |
| 总经理/董事长 | 1（全部数据） | 最高管理层 |
| 事业部总监 | 3（部门及子部门） | 管理整个事业部 |
| 部门经理 | 2（当前部门） | 管理单个部门 |
| 小组长 | 6（自己和直属下级） | 管理小团队 |
| 普通员工 | 5（仅自己） | 只看自己的数据 |
| 项目经理 | 4（自定义部门） | 跨部门协作 |

#### 配置原则

1. **最小权限原则**：默认给最小范围，按需扩大
2. **层级递减**：上级的数据范围≥下级
3. **业务优先**：根据实际业务需求选择

### 11.7 注意事项

#### ⚠️ 数据范围 ≠ 菜单权限

- **菜单权限**：控制能不能访问这个页面
- **数据范围**：控制进入页面后能看到哪些数据

**示例**：
```
某员工的角色：
  - 菜单权限：有"用户管理"菜单 ✅
  - 数据范围：仅自己（DataScope=5）

结果：
  - 能打开用户管理页面 ✅
  - 但只能看到自己创建的用户（可能为空）
```

#### ⚠️ 数据范围的优先级

**如果用户有多个角色，数据范围取并集（最宽松）**：

```
用户A的角色：
  - 技术部主管（DataScope=2，技术部）
  - 项目经理（DataScope=4，[技术部, 产品部]）

最终数据范围 = 技术部 ∪ 产品部
```

#### ⚠️ 超级管理员特殊处理

```go
if consts.IsSuperRole(user.RoleKey) {
    return model, nil  // 跳过所有过滤，看到全部数据
}
```

超级管理员**不受 DataScope 限制**，始终能看到所有数据。

### 11.8 常见问题

#### Q1：为什么配置了数据范围，还是看到所有数据？

**可能原因**：
1. 是超级管理员（自动豁免）
2. Controller 中没有应用 `FilterAuth` Handler
3. 表中没有 `dept_id` 或 `created_by` 字段

**解决方法**：
```go
// ✅ 正确：应用数据权限过滤
model := dao.YourTable.Ctx(ctx).Handler(handler.FilterAuth)

// ❌ 错误：没有应用过滤
model := dao.YourTable.Ctx(ctx)
```

#### Q2：配置"当前部门"后，为什么看不到任何数据？

**可能原因**：
1. 用户的 `dept_id` 为 0 或 NULL
2. 数据记录的 `dept_id` 字段未正确填充

**解决方法**：
- 确保用户分配了部门
- 确保数据创建时记录了部门ID

#### Q3："自己和下级"模式下，查询很慢怎么办？

**原因**：递归查询下级用户，深层级时性能差

**解决方法**（参考HotGo）：
```sql
-- 为用户表添加 tree 字段（预计算路径）
ALTER TABLE admin_user ADD COLUMN tree VARCHAR(255);

-- 查询下级变为一条SQL
SELECT * FROM admin_user WHERE tree LIKE '0,1,100,%';
```

### 11.9 数据范围配置实战指南

#### 步骤1：创建角色并设置数据范围

1. 用户管理 → 编辑用户
2. 选择角色
3. 保存

#### 步骤3：验证数据权限生效

1. 用该用户登录
2. 访问列表页面（如用户列表）
3. 查看数据是否按预期过滤

### 11.10 最佳实践建议

#### 1. 企业组织架构映射

```
公司

  │   ├─ 技术部（主管 DataScope=2）
```
